1 ParaView and Offscreen Rendering {#Offscreen}
2 ================================
3 
4 ParaView is often used to render visualization results. While in most cases,
5 the results are presented to the users on their screens or monitors, there are
6 cases when this is not required. For example, if running a batch script to
7 generate several images, one may not care to see windows flashing in front of
8 them while the results are being generated. Another use-case is when using a
9 **pvserver** or **pvrenderserver** to render images remotely and having them
10 sent back to the client for viewing. The server-side windows are do not serve
11 any useful purpose in that mode. In other cases, showing a window may not be
12 possible at all, for example on certain Linux clusters, one may not have access
13 to an X server at all, in which case on-screen rendering is just not possible
14 (this is also referred to as headless operating mode).
15 
16 This page documents the onscreen, offscreen, and headless rendering support in
17 ParaView.
18 
19 ##Caveats##
20 
21 In this page, we limit ourselves to a Linux-based system. On Windows and macOS,
22 offscreen support is available, but true headless operation is currently not
23 widely used and hence left out of this discussion.
24 
25 ##Terminology##
26 
27 A brief explanation of the terms:
28 
29 * **Desktop Qt client**: This refers to the Qt-based GUI. This is launched using
30  the **paraview** executable.
31 
32 * **Command line executables**: This refers to all other ParaView executables,
33  e.g. **pvserver**, **pvdataserver**, **pvrenderserver**, **pvpython**,
34  and **pvbatch**.
35 
36 * **Onscreen**: Onscreen refers to the case where the rendering results are
37  presented to the users in a viewable window. On Linux, this invariably requires
38  a running and accessible X server. The desktop Qt client can only operate in
39  on-screen mode and hence needs an accessible X server.
40 
41 * **Offscreen**: Offscreen simply means that the rendering results are not presented
42  to the user in a window. On Linux, this does not automatically imply that an
43  accessible X server is not needed. X server may be needed, unless ParaView
44  was built with an OpenGL implementation that allows for headless operation.
45  This mode is not applicable to the desktop Qt client and is only supported
46  by the command line executables.
47 
48 * **Headless**: Headless rendering automatically implies offscreen rendering.
49  In addition, on Linux, it implies that the rendering does not require an
50  accessible X server nor will it make any X calls.
51 
52 **Onscreen** and **offscreen** support is built by default. Thus ParaView
53 binaries available from paraview.org support these modes. **Headless** support
54 requires special builds of ParaView which are not compatible with standard
55 build options. Both version are available as a binary from our downloads page.
56 
57 ##OpenGL Implementations##
58 
59 ParaView uses OpenGL for rendering. OpenGL is an API specification for 2D/3D
60 rendering. Many vendors provide implementations of this API, including those
61 that build GPUs.
62 
63 For sake of simplicity, let's classify OpenGL implementations as **hardware
64 (H/W)** and **software (S/W)**. **H/W** includes OpenGL implementations provided by
65 NVIDIA, ATI, Intel, Apple and others which typically use the system hardware
66 infrastructure for rendering. The runtime libraries needed for these are
67 available on the system. **S/W** currently includes Mesa3D -- a software
68 implementation of the OpenGL standard. Despite the names, H/W doesn't
69 necessarily imply use of GPUs, nor does S/W imply exclusion of GPUs. Nonetheless
70 we use this naming scheme as it has been prevalent.
71 
72 ###APIs for Headless Support###
73 
74 Traditionally, OpenGL implementations are coupled with the window system to
75 provide an OpenGL context. Thus, they are designed for non-headless operation.
76 When it comes to headless operation, there are alternative APIs that an
77 application can use to create the OpenGL context that avoid this dependency on
78 the window system (or X server for the sake of this discussion).
79 
80 Currently, ParaView supports two distinct APIs that are available for headless
81 operation: **EGL** and **OSMesa** (also called **Offscreen Mesa**). It must be
82 noted that headless support is a rapidly evolving area and changes are expected
83 in coming months. Modern H/W OpenGL implementations support EGL while S/W
84 (or Mesa) supports OSMesa. One has to build ParaView with specific CMake flags
85 changed to enable either of these APIs. Which headless API you choose in your
86 build depends on which OpenGL implementation you plan to use.

88 ##ParaView Builds##
89 
90 Before we look at the various ways you can build and use ParaView, let's
91 summarize relevant CMake options available:
92 
93 * `VTK_USE_X`: When ON, implies that ParaView can link against X libraries.
94  This allows ParaView executables to create on-screen windows, if needed.
95 
96  When `VTK_USE_X` is ON, these variables must be specified:
97  * `OPENGL_INCLUDE_DIR`: Path to directory containing `GL/gl.h`.
98  * `OPENGL_gl_LIBRARY`: Path to `libGL.so`.
99  * `OpengL_glu_LIBRARY`: not needed for ParaView; leave empty.
100  * `OPENGL_xmesa_INCLUDE_DIR`: not needed for ParaView; leave empty.
101 
102 * `VTK_OPENGL_HAS_OSMESA`: When ON, implies that ParaView can use OSMesa
103  to support headless modes of operation.
104 
105  When `VTK_OPENGL_HAS_OSMESA` is ON, these variables must be specified:
106  * `OSMESA_INCLUDE_DIR`: Path to containing `GL/osmesa.h`.
107  * `OSMESA_LIBRARY`: Path to `libOSMesa.so`
108 
109 * `VTK_OPENGL_HAS_EGL`: When ON, implies that ParaView can use EGL to support
110  headless modes of operation.
111 
112  When `VTK_OPENGL_HAS_EGL` is ON, these variables must be specified:
113  * `EGL_INCLUDE_DIR`: Path to directory containing `GL/egl.h`.
114  * `EGL_LIBRARY`: Path to `libEGL.so`.
115  * `EGL_opengl_LIBRARY`: Path to `libOpenGL.so`.
116 
117 * `PARAVIEW_USE_QT` indicates if the desktop Qt client should be built.
118 
119 `VTK_USE_X` and `VTK_OPENGL_HAS_OSMESA` are mutually exclusive i.e.
120 only one of the two can be ON at the same time. This is due to the
121 fact that OSMesa does not support on-screen rendering and
122 VTK's OpenGL selection is at build time.
123 
124 `VTK_OPENGL_HAS_EGL` and `VTK_OPENGL_HAS_OSMESA` are mutually exclusive i.e.
125 only one of the two can be ON at the same time.
126 
127 A few things to note:
128 * At least one of `VTK_OPENGL_HAS_EGL`, `VTK_OPENGL_HAS_OSMESA` and `VTK_USE_X` must be ON.
129 * If `VTK_OPENGL_HAS_EGL` or `VTK_OPENGL_HAS_OSMESA` is ON, the build supports headless rendering, otherwise
130  `VTK_USE_X` must be ON and the build does not support headless, but can still support offscreen rendering.
131 * `PARAVIEW_USE_QT=ON` and `VTK_USE_X=OFF` has limited support.
132  In this configuration, the command-line tools like `pvserver`, `pvrenderserver` will not depend on X.
133  The `paraview` Qt client, however, is not built.
134 * If `VTK_OPENGL_HAS_EGL` is ON and `VTK_USE_X` is ON, then all the OpenGL and EGL variables should point
135  to the system libraries providing both, typically the NVidia libraries.

137 ##Default Rendering Modes##
138 
139 Depending of the enabled build options, which type of render window the ParaView
140 executable creates by default also needs some explanation.
141 
142 * The ParaView desktop Qt client always creates an onscreen window using GLX calls via Qt.
143 * `pvserver`, `pvrenderserver` and `pvbatch` always create an offscreen render window. If built with headless support,
144  it will be an offscreen-headless window. There are a few exceptions where it will create an onscreen window instead:
145  - if running in tile-display mode, i.e. `-tdx` or `-tdy` command line options are passed
146  - if running in immersive mode e.g. CAVE
147  - if `PV_DEBUG_REMOTE_RENDERING` environment is set
148  - if `--force-onscreen-rendering` command line option is passed.
149 * `pvpython` always creates an onscreen render window if built with onscreen support. The following are the exceptions when it
150  creates offscreen (or headless if supported) render windows.
151  - if `--force-offscreen-rendering` command line option is passed.
152 
153 `--use-offscreen-rendering` command line option supported by ParaView 5.4 and earlier has now been deprecated and is
154 interpreted as `--force-offscreen-rendering`.
