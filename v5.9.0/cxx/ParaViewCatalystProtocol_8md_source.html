1 ParaView-Catalyst Blueprint {#ParaViewCatalystBlueprint}
2 ===========================
3 
4 **This page is applicable to ParaView Catalyst implementation introduced in
5 ParaView 5.9.**
6 
7 -----
8 
9 ## Background
10 
11 Starting with ParaView 5.9, ParaView distribution provides an implementation of
12 the [Catalyst API](https://catalyst-in-situ.readthedocs.io/). This
13 implementation is now referred to as **ParaView-Catalyst**. Thus, it an
14 implementation of the Catalyst In Situ API that uses ParaView for data
15 processing and rendering.
16 
17 The Catalyst in situ API comprises for 3 main function calls that are used to
18 pass data and control over to the Catalyst implementation from a computational
19 simulation codes: `catalyst_initialize`, `catalyst_execute`, and
20 `catalyst_finalize`. Each of these functions is passed a
21 [Conduit Node](https://llnl-conduit.readthedocs.io/en/latest/tutorial_cpp_basics.html)
22 object. Conduit Node provides a flexible mechanism for describing hierarchical
23 in-core scientific data. Since Conduit Node is simply a light-weight container,
24 we need to define the conventions used to communicate relevant data in each of
25 the Catalyst API calls. Such conventions are collectively referred to as the
26 Blueprint. Each Catalyst API implementation can develop its own blueprint.
27 ParaView-Catalyst, the Catalyst API implementation provided by ParaView, also
28 defines its blueprint called the **ParaView-Catalyst Blueprint**. This page
29 documents this blueprint.
30 
31 ## Protocol
32 
33 A blueprint often includes several protocols, each defining the conventions for
34 a specific use-case. Since there are three main Catalyst API functions, the
35 blueprint currently defines three protocols, one for each of the Catalyst
36 API functions.
37 
38 * **protocol: 'initialize'**: defines the options accepted by
39  `catalyst_initialize`; these include things like ParaView Python scripts to
40  load.
41 
42 * **protocol: 'execute'**: defines the protocol for `catalyst_execute` and
43  includes information about Catalyst channels i.e. ports on which data is made
44  available to in situ processing as well as the actual data from the
45  simulation.
46 
47 * **protocol: 'finalize'**: defines the protocol for `catalyst_finalize`;
48  currently, this is empty.
49 
50 
51 In each of the Catalyst API calls, ParaView looks for a top-level node named
52 'catalyst'. The expected children vary based on the protocol described in the
53 following sub-sections.
54 
55 These top-level protocols use other internal protocols e.g. 'channel'.
56 
57 ### protocol: 'initialize'
58 
59 Currently, 'initialize' protocol defines how to pass scripts to load for
60 analysis.
61 
62 * catalyst/scripts: (optional) if present must either be a 'list' or 'object'
63  node with child nodes that provides paths to the Python scripts to load for
64  in situ analysis.
65 
66 Additionally, one can provide a list of pre-compiled pipelines to use.
67 
68 * catalyst/pipelines: (optional) if present must be a 'list' or 'object' node
69  with child nodes that are objects that provide type and parameters for
70  hard-coded pipelines to use. Each object must be in accordance to the
71  protocol: 'pipeline'.
72 
73 In MPI-enabled builds, ParaView is by default initialized to use `MPI_COMM_WORLD`
74 as the global communicator. A specific MPI communicator can be provided as
75 follows:
76 
77 * catalyst/mpi\_comm: (optional) if present, must be an integer representing the
78 Fortran handle for the MPI communicator to use. The Fortran handle can be
79 obtained from `MPI_Comm` using `MPI_Comm_c2f()`.
80 
81 ### protocol: 'execute'
82 
83 Defines now to communicate data during each time-iteration.
84 
85 **time/timestep/cycle**: this defines temporal information about the current
86 invocation.
87 
88 * catalyst/state/timestep: (optional) integral value for current timestep. if not
89  specified, 'catalyst/cycle' is used.
90 * catalyst/state/cycle: (optional) integral value for current cycle index, if not
91  specified, 0 is assumed.
92 * catalyst/state/time: (optional) float64 value for current time, if not specified,
93  0.0 is assumed.
94 
95 **channels**: channels are used to communicate simulation data. The **channels**
96 node can have one or more children, each corresponding to a named channel. A
97 channel represents a data-source in the analysis pipeline that is linked to the
98 data being produced by the simulation code.
99 
100 * catalyst/channels/[channel-name]: (protocol: 'channel'): (optional) if present
101  represents a named channel with the name 'channel-name'. The node must be an
102  'object' node satisfying the 'channel' protocol.
103 
104 The 'channel' protocol is as follows:
105 
106 * channel/type: (required) a string representing the channel type. Currently,
107  the only supported value is "mesh". It is used to indicate that this channel
108  is specified in accordance to the
109  [Conduit Mesh](https://llnl-conduit.readthedocs.io/en/latest/blueprint_mesh.html#) protocol.
110 
111 * channel/data: (required) an object node used to communicate the simulation
112  data on this channel. This node must match the protocol requirements
113  identified by the 'channel/type'.
114 
115 ### protocol: 'finalize'
116 
117 Currently, this is empty.
118 
119 ### protocol: 'pipeline'
120 
121 Defines type and parameters for a hard-coded pipeline.
122 
123 * type: (required) a string identifying the type of the pipeline. Currently
124  supported value is "io".
125 
126 When 'type' is 'io', the following attributes are supported.
127 
128 * filename: (required) a string representing a filename. `%ts` may be used to
129  replace with timestep or `%t` may be used to replace with time value. You can
130  use `printf` style format specifiers e.g. `%03t` etc. The filename may be used
131  to determine which supported writer to use for saving the data.
132 
133 * channel: (required) a string identifying the channel by its name.
