1 Major API Changes {#MajorAPIChanges}
2 =================
3 
4 This page documents major API/design changes between different versions since we
5 started tracking these (starting after version 4.2).
6 
7 Changes in 5.5
8 --------------
9 
10 ###Changes to Python shell###
11 
12 `pqPythonShell` now is a fairly self contained class. One no longer has to go
13 through `pqPythonManager` or `pqPythonDialog` to create it. `pqPythonDialog` has
14 been deprecated. `pqPythonManager` has been relegated to a helper class to
15 support macros (together with `pqPythonMacroSupervisor`). The Python macro
16 infrastructure may be revamped in future, but is left unchanged for now.
17 
18 If you want a Python shell in your application, simply create an instance of
19 `pqPythonShell` and add it where ever you choose. Multiple instances are also
20 supported.
21 
22 ###Changes to offscreen rendering options###
23 
24 ParaView executables now automatically choose to create on-screen or off-screen
25 render windows for rendering based on the executable type and current
26 configuration. As a result **"UseOffscreenRendering"** property on views has
27 been removed to avoid conflicting with the new approach. Executables have new
28 command line arguments: `--force-offscreen-rendering` and
29 `--force-onscreen-rendering` that can be used to override the default behavior
30 for the process.
31 
32 ###Changes to vtkArrayCalculator
33 `vtkArrayCalculator::SetAttributeMode` is deprecated in favor of `vtkArrayCalculator::SetAttributeType`
34 which takes vtkDataObject attribute modes instead of custom constants as its parameter value.
35 The ParaView calculator filter's AttributeMode property's values changed as a result of this.
36 
37 ###Changes to vtkSMViewProxy::CaptureWindowInternal###
38 
39 `vtkSMViewProxy::CaptureWindowInternal` now takes a 2-component magnification
40 factor rather than a single component. That allows for more accurate target image
41 resolution than before (see #17567).
42 
43 ###Replaced LockScalarRange property in PVLookupTable with AutomaticRescaleRangeMode###
44 
45 The `LockScalarRange` property in the `PVLookupTable` proxy has been removed. It
46 has been replaced by the `AutomaticRescaleRangeMode` property. When
47 `AutomaticRescaleRangeMode` is set to `vtkSMTransferFunctionManager::Never`,
48 the transfer function minimum and maximum value is never updated, no matter what
49 event occurs. This corresponds to when `LockScalarRange` was set to "on" in
50 previous versions of ParaView. When `AutomaticRescaleRangeMode` is set to a
51 different option, that option governs how and when the transfer function is
52 reset. This option overrides whichever option is set in the `GeneralSettings`
53 property `TransferFunctionResetMode`.
54 
55 The `TransferFunctionResetMode` property still exists, but it has been slightly
56 repurposed. No longer does it control the range reset mode for ALL transfer
57 functions. Instead, it serves the following functions:
58 
59 * determining the `AutomaticRescaleRangeMode` when a lookup table is created, and
60 
61 * determining what the `AutomaticRescaleRangeMode` should be when the data array
62  range is updated to the data range of the selected representation
63 
64 As part of this change, the enum values
65 
66  GROW_ON_APPLY
67  GROW_ON_APPLY_AND_TIMESTEP
68  RESET_ON_APPLY
69  RESET_ON_APPLY_AND_TIMESTEP
70 
71 have been moved from `vtkPVGeneralSettings.h` to `vtkSMTransferFunctionManager.h`.
72 
73 ###Deprecated pqDisplayPolicy###
74 
75 `pqDisplayPolicy`, a class that has been unofficially deprecated since
76 `vtkSMParaViewPipelineControllerWithRendering` was introduced is now officially
77 deprecated and will be removed in future. Use
78 `vtkSMParaViewPipelineControllerWithRendering` to show/hide data in views
79 instead of `pqDisplayPolicy`.
80 
81 ###Changes to pqParaViewMenuBuilders###
82 
83 The signature of `pqParaViewMenuBuilders::buildPipelineBrowserContextMenu(QWidget&)`
84 has changed to `pqParaViewMenuBuilders::buildPipelineBrowserContextMenu(QMenu&)` and
85 now requires a `QMenu` to populate rather than populating the implicitly created context
86 menu in a widget.
87 
88 Note that the `contextMenuPolicy` set in the widget providing the context menu must be
89 set to `Qt::DefaultContextMenu` for the context menu to appear. in `pqPipelineBrowserWidget`,
90 this used to be set to `Qt::ActionsContextMenu`.

92 Changes in 5.4
93 --------------
94 
95 ###Moved vtkAppendArcLength to VTK and exposed this filter in the UI.
96 
97 As a result of this change, `vtkAppendArcLength` was moved from
98 the `internal_filters` to the `filters` group.

100 ###Changes to Save Screenshot / Save Animation###
101 
102 `pqSaveSnapshotDialog` is removed. This class was used by
103 `pqSaveScreenshotReaction` to present user with options to control the saved
104 image. This is now done using `pqProxyWidgetDialog` for the **(misc,
105 SaveScreenshot)** proxy. See `pqSaveScreenshotReaction` for details.
106 
107 `pqStereoModeHelper` has been removed. It was to help save/restore stereo mode
108 for views for saving screenshots and animations. All of that logic is now
109 encapsulated in `vtkSMSaveScreenshotProxy`.
110 
111 `pqMultiViewWidget::captureImage`, `pqMultiViewWidget::writeImage`,
112 `pqTabbedMultiViewWidget::captureImage`, `pqTabbedMultiViewWidget::writeImage`,
113 `pqView::captureImage`, `pqView::writeImage` have been deprecated and will be removed in
114 future releases. Saving and capturing images now goes through pqSaveScreenshotReaction
115 (or **vtkSMScreenshotOptions** proxy).

117 Changes in 5.3

119 
120 ###Removed pqRescaleCustomScalarRangeReaction, pqRescaleVisibleScalarRangeReaction###
121 
122 `pqRescaleCustomScalarRangeReaction` and `pqRescaleVisibleScalarRangeReaction`
123 have been removed and replaced by the more
124 generic `pqResetScalarRangeReaction`. Set mode
125 to`pqResetScalarRangeReaction::CUSTOM` when instantiating
126 `pqResetScalarRangeReaction` to make it behave as
127 `pqRescaleCustomScalarRangeReaction`. The mode can be set to
128 `pqResetScalarRangeReaction::VISIBLE`, for
129 `pqRescaleVisibleScalarRangeReaction`.
130 
131 ###Dropped support for pqProxyPanel and subclasses (Legacy Panels)###
132 
133 `pqProxyPanel` and its subclasses that formed the default mechanism for providing
134 custom panels for filters, displays, views, etc. before ParaView 3.98 has now
135 been removed. Refer to the
136 [wiki](http://www.paraview.org/Wiki/ParaView/Properties_Panel) for getting
137 oriented with the currently supported way for customizing the properties panel.
138 This also removes plugin macros `ADD_PARAVIEW_OBJECT_PANEL()`, and
139 `ADD_PARAVIEW_DISPLAY_PANEL()`.

141 ###Qt 5 Support (replacing QVTKWidget)###
142 
143 ParaView has switched over to using Qt 5 by default. While for most cases the
144 application code should be unaffected, if your app was directly creating (or
145 using) `QVTKWidget` however, you will need to change it. When
146 building with Qt 5, you are expected to use `QVTKOpenGLWidget` instead of
147 `QVTKWidget` (which is still needed when building with Qt 4).
148 Since ParaView supports building with Qt 4 and 5 for this release, to make it
149 easier, we have the following solution.
150 
151 A new header **pqQVTKWidgetBase.h** defines a new typedef `pqQVTKWidgetBase`.
152 Include this header in your code and replace `QVTKWidget` with `pqQVTKWidgetBase`.
153 `pqQVTKWidgetBase` gets typedef'ed to `QVTKWidget` for Qt 4 builds and to
154 `QVTKOpenGLWidget` for Qt 5 builds.
155 
156 If your existing code was creating and setting up a new instance of
157 `QVTKWidget`, you need to change it further. QVTKOpenGLWidget cannot work with
158 any `vtkRenderWindow` (as was the case with QVTKWidget),
159 it instead needs a `vtkGenericRenderWindow` (see the documentation
160 for `QVTKOpenGLWidget` for details). To make that easier, **pqQVTKWidgetBase.h**
161 defines another typedef `pqQVTKWidgetBaseRenderWindowType`. The code to create
162 and setup QVTKWidget can then be changed as follows:
163 
164  vtkNew<pqQVTKWidgetBaseRenderWindowType> renWindow;
165  pqQVTKWidgetBase* widget = new pqQVTKWidgetBase(...);
166  widget->SetRenderWindow(renWindow.Get());
167 
168 If you are using a `vtkView` to create the viewport. You can set that up as
169 follows:
170 
171  vtkNew<pqQVTKWidgetBaseRenderWindowType> renWindow;
172  pqQVTKWidgetBase* widget = new pqQVTKWidgetBase(...);
173  widget->SetRenderWindow(renWindow.Get());
174 
175  vtkNew<vtkContextView> view;
176  view->SetRenderWindow(renWindow.Get());
177 
178 If your application supports Qt 5 alone, when you should directly use
179 `QVTKOpenGLWidget`. `pqQVTKWidgetBase` is a temporary solution to support
180 building with Qt 5 and Qt 4. Qt 4 support will be deprecated and remove in
181 future releases.

188 Changes in 5.2
189 --------------
190 
191 ###Removed pqStandardArrayColorMapsBehavior###
192 
193 Setting the default colormaps from a Qt Behavior was causing inconsistent results
194 between the ParaView UI and pvpython/pvbatch. Adding these default settings was
195 moved into the server manager. To override these defaults, use
196 vtkSMSettings::AddCollectionFromX with a priority greater than 0. The settings
197 affected are the default colormap for arrays with names vtkBlockColors and
198 AtomicNumbers. With these settings moved, the Qt behavior
199 pqStandardArrayColorMapsBehavior no longer does anything and so it has been removed.
200 
201 ###Qt dependencies###
202 
203 Starting with 5.2, ParaView natively supports Qt 4 and Qt 5. To simplify writing
204 code with either dependency, we now have a new CMake file `ParaViewQt.cmake`
205 that gets included by `PARAVIEW_USE_FILE`. This provides new macros that be used
206 to find qt (`pv_find_package_qt`), wrap cpp (`pv_qt_wrap_cpp`), ui
207 (`pv_qt_wrap_ui`), or add resources (`pv_qt_add_resources`) instead of using
208 `qt4_` or `qt4_` specific versions based on whether the app is being built with
209 Qt4 or Qt5. `pv_find_package_qt` accepts optional `QT5_COMPONENTS` and
210 `QT4_COMPONENTS` which can be used to list the Qt component dependencies for
211 each of the versions. e.g.
212 
213  include(ParaViewQt) # generally not needed, since auto-included
214  pv_find_package_qt(qt_targets
215  QT4_COMPONENTS QtGui
216  QT5_COMPONENTS Widgets)
217 
218  pv_qt_wrap_cpp(moc_files ${headers})
219  pv_qt_wrap_ui(ui_files ${uis})
220 
221  ...
222  target_link_libraries(${target} LINK_PRIVATE ${qt_targets})
223 
224 ###Multiple input ports with vtkPythonProgrammableFilter###
225 
226 vtkPythonProgrammableFilter can now accept multiple input ports if the number
227 of input ports is defined in the XML plugin file with the *input_ports* attribute.
228 The different input ports are then defined with InputProperty having each a
229 different *port_index*:
230 
231  <SourceProxy name="Name" class="vtkPythonProgrammableFilter" label="label" input_ports="2">
232  <InputProperty name="Source" command="SetInputConnection" port_index="0">
233  [...]
234  </InputProperty>
235  <InputProperty name="Target" command="SetInputConnection" port_index="1">
236  [...]
237  </InputProperty>
238  </SourceProxy>

240 Changes in 5.1
241 --------------
242 
243 ###Removed Cube Axes###
244 
245 Cube axes, including support in UI, Python as well as the related ParaView
246 specific classes e.g. `vtkCubeAxesRepresentation`, `pqCubeAxesEditorDialog`, and
247 `pqCubeAxesPropertyWidget` have been removed. The cube
248 axes was replaced by a generally preferred axes annotation implementation called
249 **Axes Grid**. While the two are not compatible (API-wise or visually), using
250 Axes Grid generates a more pleasing visualization.
251 
252 ###Removed `pqWriterDialog`###
253 
254 `pqWriterDialog` was used by `pqSaveDataReaction` to show a dialog for the user to
255 change the writer's properties. However, since the class was added, we have
256 created a new, more generic, pqProxyWidgetDialog that also allows the user to
257 save his choices to the application settings. `pqSaveDataReaction` now simply
258 uses `pqProxyWidgetDialog` instead of `pqWriterDialog`. `pqWriterDialog` class
259 has been removed.
260 
261 ###Refactored 3DWidget panels###
262 
263 3DWidget panels were subclasses of `pqObjectPanel`, the Properties panel
264 hierarchy that has been deprecated since 4.0. This version finally drops support
265 for these old 3D widget panels. The new implementation simply uses the
266 `pqPropertyWidget` infrastructure. A 3D widget panel is nothing more than a custom
267 `pqPropertyWidget` subclass for a property-group (`vtkSMPropertyGroup`) which
268 creates an interactive widget to be shown in the active view to help the users
269 change the values for properties in the property-group using this interactive
270 widget, in addition to standard Qt-based UI elements.
271 
272 If you have a filter or proxy indicating that the Properties panel use one of
273 the standard 3D widgets for controlling certain properties on that proxy, you
274 specified that using `<Hints>` in the Proxy's XML definition.
275 
276 For example, the **(implicit_functions, Plane)** proxy requested that
277 `pqImplicitPlaneWidget` be used using the following XML hints.
278 
279  <Proxy class="vtkPVPlane" name="Plane">
280  <InputProperty is_internal="1" name="Input" />
281  <DoubleVectorProperty command="SetOrigin"
282  default_values="0.0 0.0 0.0"
283  name="Origin"
284  number_of_elements="3">
285  ...
286  </DoubleVectorProperty>
287  <DoubleVectorProperty command="SetNormal"
288  default_values="1.0 0.0 0.0"
289  name="Normal"
290  number_of_elements="3">
291  ...
292  </DoubleVectorProperty>
293  <DoubleVectorProperty animateable="1"
294  command="SetOffset"
295  default_values="0.0"
296  name="Offset"
297  number_of_elements="1">
298  </DoubleVectorProperty>
299  <Hints>
300  <PropertyGroup type="Plane">
301  <Property function="Origin" name="Origin" />
302  <Property function="Normal" name="Normal" />
303  </PropertyGroup>
304  <ProxyList>
305  <Link name="Input" with_property="Input" />
306  </ProxyList>
307  </Hints>
308  </Proxy>
309 
310 Since 3D widget panels are now simply custom `pqPropertyWidget` subclasses for a
311 property-group, this code changes are follows:
312 
313  <Proxy class="vtkPVPlane" name="Plane">
314  <InputProperty is_internal="1" name="Input" />
315  <DoubleVectorProperty command="SetOrigin"
316  default_values="0.0 0.0 0.0"
317  name="Origin"
318  number_of_elements="3">
319  ...
320  </DoubleVectorProperty>
321  <DoubleVectorProperty command="SetNormal"
322  default_values="1.0 0.0 0.0"
323  name="Normal"
324  number_of_elements="3">
325  ...
326  </DoubleVectorProperty>
327  <DoubleVectorProperty animateable="1"
328  command="SetOffset"
329  default_values="0.0"
330  name="Offset"
331  number_of_elements="1">
332  </DoubleVectorProperty>
333  <PropertyGroup label="Plane Parameters" panel_widget="InteractivePlane">
334  <Property function="Origin" name="Origin" />
335  <Property function="Normal" name="Normal" />
336  <Property function="Input" name="Input" />
337  </PropertyGroup>
338  <Hints>
339  <ProxyList>
340  <Link name="Input" with_property="Input" />
341  </ProxyList>
342  </Hints>
343  </Proxy>
344

346 
347 - The `<PropertyGroup>` tag is no longer specified under `<Hints>`.
348 - The `panel_widget` attribute is used to indicate which custom widget to create
349 for this property group.

351 The functions for properties in the group supported by each type 3D widget (or
352 interactive widget as we now will call them) can be found by looking at the

368 ###Changes to vtkDataArray Subclasses and In-Situ Arrays###
369 
370 vtkDataArray can now support alternate memory layouts via the
371 vtkAOSDataArrayTemplate or other subclasses of
372 vtkGenericDataArray. While the older (and slower) vtkTypedDataArray
373 and vtkMappedDataArray interfaces still exist, they should not be used
374 and direct vtkGenericDataArray subclasses are preferred. See
375 http://www.vtk.org/Wiki/VTK/Tutorials/DataArrays for more details on
376 working with the new arrays.
377 
378 The following methods have been deprecated in vtkAOSDataArray:
379 
380 ~~~{.cpp}
381 void GetTupleValue(vtkIdType tupleIdx, ValueType *tuple)
382 void SetTupleValue(vtkIdType tupleIdx, const ValueType *tuple)
383 void InsertTupleValue(vtkIdType tupleIdx, const ValueType *tuple)
384 vtkIdType InsertNextTupleValue(const ValueType *tuple)
385 ~~~
386 
387 The replacements use `TypedTuple` in place of `TupleValue`.

389 Changes in 5.0.1
390 ----------------
391 
392 ###Changes to vtkSMInputArrayDomain###
393 
394 vtkSMInputArrayDomain has changed the meaning of **"any"** attribute type. It
395 now includes field data arrays. To exclude field data arrays from the field
396 selection, change this to **"any-except-field"**. This is also the default for
397 vtkSMInputArrayDomain, hence simply removing the `attribute_type` field is also
398 an acceptable solution and is the recommended approach so that the XML can work
399 without changes in earlier versions of ParaView as well.

401 Changes in 5.0
402 --------------
403 
404 ###Changed *Source* property to **GlyphType** on certain representations###
405 
406 *Source* property on representations, which was not exposed in the UI has been
407 changed to *GlyphType* to better match its role. Since this property was not
408 exposed earlier, this change should not affect any users except custom
409 applications that explicitly used this property.
410 
411 
412 ###Changes to caching infrastructure in **vtkPVDataRepresentation** and subclasses###
413 
414 To avoid extra work and sync issues when propagating **UseCache** and
415 **CacheKey** flags from a vtkPVView to vtkPVDataRepresentation,
416 vtkPVDataRepresentation was updated to directly obtain thet values for these
417 flags from the view. To enable this, vtkPVDataRepresentation saves a weak
418 reference to the View it's being added to in vtkPVDataRepresentation::AddToView
419 and vtkPVDataRepresentation::RemoveFromView. Subclasses of
420 vtkPVDataRepresentation didn't consistently call the superclass implementation,
421 hence any representation subclass should ensure that it calls the superclass
422 implementations of AddToView and RemoveFromView for this to work correctly.
423 vtkPVView::AddRepresentationInternal checks for this and will raise a runtime
424 error if a representaiton is encountered that didn't properly set its View.
425 
426 The above change also makes **vtkPVDataRepresentation::SetUseCache** and
427 **vtkPVDataRepresentation::SetCacheKey** obsolete. Subclasses no longer need to
428 provide any implementation for these methods and they should simply be removed.

430 Changes in 4.4
431 --------------
432 
433 ###Refactored pqView widget creation mechanisms###
434 
435 pqView now adds a pure virtual method pqView::createWidget(). Subclasses should
436 override that method to create the QWidget in which the view is *rendered*. In
437 the past the subclasses implemented their own mechanisms to create the widget
438 and return it when `getWidget()` was called the first time.
439 
440 `pqView::getWidget()` is now deprecated. Users should use pqView::widget()
441 instead. This method internally calls the pqView::createWidget() when
442 appropriate.
443 
444 ###Removed vtkPVGenericRenderWindowInteractor, vtkPVRenderViewProxy###
445 
446 ParaView was subclassing vtkRenderWindowInteractor to create
447 `vtkPVGenericRenderWindowInteractor` to handle interaction. That piece of code
448 was potentially derived from an older implementation of
449 vtkRenderWindowInteractor and hence did what it did. Current implementation of
450 vtkRenderWindowInteractor lets the vtkInteractionStyle (and subclasses) do all
451 the heavy lifting. ParaView did that to some extent (since it has a
452 vtkPVInteractorStyle), but will was relying on
453 `vtkPVGenericRenderWindowInteractor`, `vtkPVRenderViewProxy` to propagate
454 interaction/still renders and other things. This has been refactored. ParaView
455 no longer uses a special vtkRenderWindowInteractor. All logic is handled by
456 observers on the standard vtkRenderWindowInteractor.
457 
458 This change was done to make it easier to enable interactivity in `pvpython`.
459 
460 See also: vtkSMRenderViewProxy::SetupInteractor(). Subclasses of pqView now pass
461 the interactor created by QVTKWidget to this method to initialize it.
462 
463 See also: vtkSMViewProxyInteractorHelper.
464 
465 ###Refactored Color Preset Management (and UI)###
466 
467 The color preset management has been completely refactored for this release.
468 This makes presets accessible to Python clients.
469 `vtkSMTransferFunctionPresets` is the class that manages the presets in the
470 *ServerManager* layer.
471 
472 On the UI side, `pqColorPresetModel`, `pqColorPresetManager`, and
473 `pqColorMapModel` no longer exist. They have been replaced by `pqPresetDialog`.
474 Since the UI is completely redesigned, tests in custom applications
475 that used the color preset dialog will need to be updated as well.
476 
477 The preset themselves are now serialized as JSON in the same form as the
478 settings JSON. ColorMaps in legacy XML format are still loadable from the UI. At
479 the same time, a tool `vtkLegacyColorMapXMLToJSON` is available to convert such
480 XMLs to JSON.

494 
495 `vtkSMProxyIterator::Begin(const char* groupName)` now additionally

502 Changes in 4.3
503 --------------
504 
505 ###Replaced `pqCurrentTimeToolbar` with `pqAnimationTimeWidget`###
506 
507 `pqCurrentTimeToolbar` was a `QToolbar` subclass designed to be used in an
508 application to allow the users to view/change the animation time. We
509 reimplemented this class as `pqAnimationTimeWidget` making it a `QWidget`
510 instead of `QToolbar` to make it possible to use that in other widgets e.g.
511 `pqTimeInspectorWidget`. `pqAnimationTimeToolbar` was also changed to use
512 `pqAnimationTimeWidget` instead of `pqCurrentTimeToolbar`.
513 
514 Besides just being a cleaner implementation (since it uses `pqPropertyLinks`
515 based linking between ServerManager and UI), `pqAnimationTimeWidget` also allows
516 applications to change the animation *play mode*, if desired. The rest of the
517 behavior of this class is similar to `pqCurrentTimeToolbar` and hence should not
518 affect applications (besides need to update any tests since the widget names
519 have now changed).
520 
521 ###Removed the need for plugins to specify a GUI_RESOURCE_FILE###
522 
523 It is cleaner and easier to update one xml file for each plugin.
524 Hints in the server manager xml file already perform the same functionality and
525 the GUI xml file was deprecated. Sources are now added to the **Sources** menu
526 and filters are added to the **Filters** menu when they are detected in the server
527 xml file. To place sources/filters in a category, add the following hint to the
528 Proxy XML element (`category` and `icon` attributes are optional):
529 
530  <SourceProxy ...>
531  ...
532  <Hints>
533  <ShowInMenu category="<category-name>" icon="<qt-image-resource>" />
534  </Hints>
535  </SourceProxy>
536 
537 The ability to add filters to the sources menu (as done by a few plugins) is
538 removed by this change.
539 
540 This change also modifies how readers are detected. Readers now must provide a
541 hint to the ReaderManager in the server xml file to be detected as readers rather
542 than sources. The backwards compatibility behavior that assumed any source with
543 a FileName attribute was a reader has been removed.
544 
545 ###Removed pqActiveView (use pqActiveObjects instead)###
546 
547 `pqActiveView` was a long deprecated class which internally indeed used
548 pqActiveObjects. `pqActiveView` has now been removed. Any code using
549 `pqActiveView` can switch to using pqActiveObjects with very few code changes.
