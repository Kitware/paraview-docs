1 Develop ParaView with Git
2 =========================
3 
4 This page documents how to develop ParaView through [Git][].
5 See the [README](README.md) for more information.
6 
7 [Git]: http://git-scm.com
8 
9 Git is an extremely powerful version control tool that supports many
10 different "workflows" for individual development and collaboration.
11 Here we document procedures used by the ParaView development community.
12 In the interest of simplicity and brevity we do *not* provide an
13 explanation of why we use this approach.
14 
15 Setup
16 -----
17 
18 Before you begin, perform initial setup:
19 
20 1. Register [GitLab Access][] to create an account and select a user name.
21 
22 2. [Fork ParaView][] into your user's namespace on GitLab.
23 
24 3. Follow the [download instructions](download.md#clone) to create a
25  local clone of the main ParaView repository. Optionally configure
26  Git to [use SSH instead of HTTPS](download.md#use-ssh-instead-of-https).
27  Then clone:
28 
29  $ git clone --recursive https://gitlab.kitware.com/paraview/paraview.git ParaView
30  $ cd ParaView
31  The main repository will be configured as your `origin` remote.
32 
33 4. Run the [developer setup script][] to prepare your ParaView work tree and
34  create Git command aliases used below:
35 
36  $ ./Utilities/SetupForDevelopment.sh
37  This will prompt for your GitLab user name and configure a remote
38  called `gitlab` to refer to it.
39 
40 5. (Optional but highly recommended.)
41  [Register](https://open.cdash.org/register.php) with the ParaView project
42  on Kitware's CDash instance to better know how your code performs in
43  regression tests. After registering and signing in, click on
44  "All Dashboards" link in the upper left corner, scroll down and click
45  "Subscribe to this project" on the right of ParaView.
46

48 [Fork ParaView]: https://gitlab.kitware.com/paraview/paraview/forks/new
49 [developer setup script]: /Utilities/SetupForDevelopment.sh

51 Workflow
52 --------
53 
54 ParaView development uses a [branchy workflow][] based on topic branches.
55 Our collaboration workflow consists of three main steps:
56 
57 1. Local Development:
58  * [Update](#update)
59  * [Create a Topic](#create-a-topic)
60 
61 2. Code Review (requires [GitLab Access][]):
62  * [Share a Topic](#share-a-topic)
63  * [Create a Merge Request](#create-a-merge-request)
64  * [Review a Merge Request](#review-a-merge-request)
65  * [Revise a Topic](#revise-a-topic)
66 
67 3. Integrate Changes:
68  * [Merge a Topic](#merge-a-topic) (requires permission in GitLab)
69  * [Delete a Topic](#delete-a-topic)
70 
71 [branchy workflow]: http://public.kitware.com/Wiki/Git/Workflow/Topic
72 
73 Update
74 ------
75 
76 1. Update your local `master` branch:
77 
78  $ git checkout master
79  $ git pullall
80 2. Optionally push `master` to your fork in GitLab:
81 
82  $ git push gitlab master
83  to keep it in sync. The `git gitlab-push` script used to
84  [Share a Topic](#share-a-topic) below will also do this.
85 
86 Create a Topic
87 --------------
88 
89 All new work must be committed on topic branches.
90 Name topics like you might name functions: concise but precise.
91 A reader should have a general idea of the feature or fix to be developed given
92 just the branch name. Additionally, it is preferred to have an issue associated with
93 every topic. The issue can document the bug or feature to be developed. In such
94 cases, being your topic name with the issue number.
95 
96 1. To start a new topic branch:
97 
98  $ git fetch origin
99 
100  If there is an issue associated with the topic, assign the issue to yourself
101  using the "**Assignee**" field, and add the
102  `workflow:active-development` label to it.
103 
104 2. For new development, start the topic from `origin/master`:
105 
106  $ git checkout -b my-topic origin/master
107 
108  For release branch fixes, start the topic from `origin/release`, and
109  by convention use a topic name starting in `release-`:
110 
111  $ git checkout -b release-my-topic origin/release
112 
113  If subdmodules may have changed, the run:
114 
115  $ git submodule update
116 
117 3. Edit files and create commits (repeat as needed):
118 
119  $ edit file1 file2 file3
120  $ git add file1 file2 file3
121  $ git commit
122 
123  Caveats:
124  * To add data follow these [vtk instructions][].
125 
126  Commit messages must contain a brief description as the first line
127  and a more detailed description of what the commit contains. If
128  the commit contains a new feature, the detailed message must
129  describe the new feature and why it is needed. If the commit
130  contains a bug fix, the detailed message must describe the bug
131  behavior, its underlying cause, and the approach to fix it. If the
132  bug is described in the bug tracker, the commit message must
133  contain a reference to the bug number.
134 
135 4. Add some tests
136 
137  * Start `paraview.exe -dr` to ignore prefs (disable registry)
138  * Choose `Tools .. Record Test` to start.
139  * Choose `Tools .. Lock View Size Custom...` - a 400x400 window works well.
140  * Perform actions in the GUI that exercise your feature. Stop recording.
141  * Put the resulting XML file into `Applications/ParaView/Testing/XML`
142  * Add it to CMakeLists.txt, probably in a TESTS_WITH_BASELINES section
143  * you can manually add `<pqcompareview>` for multiple image comparisons, then add to the TESTS_WITH_INLINE_COMPARES section
144  * Follow the [vtk instructions][] to add the baseline images, which live in `Testing/Data/Baseline/`.
145  * Add all testing files to your topic.
146 
147  Some background is in the [testing design wiki](https://www.paraview.org/Wiki/Testing_design).
148 
149 5. Add release notes
150 
151  Notable changes should create a new file in `Documentation/release/dev/`
152  named `my-topic.md` (replace `my-topic` with the name of your topic).
153  This is not necessary for branches which are "trivial" such as fixing
154  typos, updating test baselines, or are developer-oriented.
155

158 Share a Topic
159 -------------
160 
161 When a topic is ready for review and possible inclusion, share it by pushing
162 to a fork of your repository in GitLab. Be sure you have registered and
163 signed in for [GitLab Access][] and created your fork by visiting the main
164 [ParaView GitLab][] repository page and using the "Fork" button in the upper right.
165 
166 [ParaView GitLab]: https://gitlab.kitware.com/paraview/paraview
167 
168 1. Checkout the topic if it is not your current branch:
169 
170  $ git checkout my-topic
171 
172 2. Check what commits will be pushed to your fork in GitLab:
173 
174  $ git prepush
175 
176 3. Push commits in your topic branch to your fork in GitLab:
177 
178  $ git gitlab-push
179 
180  Notes:
181  * If you are revising a previously pushed topic and have rewritten the
182  topic history, add `-f` or `--force` to overwrite the destination.
183  * If the topic adds data see [this note](https://gitlab.kitware.com/vtk/vtk/blob/master/Documentation/dev/git/data.md#push).
184  * The `gitlab-push` script also pushes the `master` branch to your
185  fork in GitLab to keep it in sync with the upstream `master`.
186 
187  The output will include a link to the topic branch in your fork in GitLab
188  and a link to a page for creating a Merge Request.

190 Create a Merge Request
191 ----------------------
192 
193 (If you already created a merge request for a given topic and have reached
194 this step after revising it, skip to the [next step](#review-a-merge-request).)
195 
196 Visit your fork in GitLab, browse to the "**Merge Requests**" link on the
197 left, and use the "**New Merge Request**" button in the upper right to
198 reach the URL printed at the end of the [previous step](#share-a-topic).
199 It should be of the form:
200 
201  https://gitlab.kitware.com/<username>/paraview/merge_requests/new
202 
203 Follow these steps:
204 
205 1. In the "**Source branch**" box select the `<username>/paraview` repository
206  and the `my-topic` branch.
207 
208 2. In the "**Target branch**" box select the `paraview/paraview` repository and
209  the `master` branch. It should be the default.
210 
211  If your change is a fix for the `release` branch, you should still
212  select the `master` branch as the target because the change needs
213  to end up there too.
214 
215 3. Use the "**Compare branches**" button to proceed to the next page
216  and fill out the merge request creation form.
217 
218 4. In the "**Title**" field provide a one-line summary of the entire
219  topic. This will become the title of the Merge Request.
220 
221  Example Merge Request Title:
222 
223  Wrapping: Add Java 1.x support
224 
225 5. In the "**Description**" field provide a high-level description
226  of the change the topic makes and any relevant information about
227  how to try it.
228  * Use `@username` syntax to draw attention of specific developers.
229  This syntax may be used anywhere outside literal text and code
230  blocks. Or, wait until the [next step](#review-a-merge-request)
231  and add comments to draw attention of developers.
232  * If your change is a fix for the `release` branch, indicate this
233  so that a maintainer knows it should be merged to `release`.
234  * Optionally use a fenced code block with type `message` to specify
235  text to be included in the generated merge commit message when the
236  topic is [merged](#merge-a-topic).
237 
238  Example Merge Request Description:
239 
240  This branch requires Java 1.x which is not generally available yet.
241  Get Java 1.x from ... in order to try these changes.
242 
243  ```message
244  Add support for Java 1.x to the wrapping infrastructure.
245  ```
246 
247  Cc: @user1 @user2
248 
249 6. The "**Assign to**", "**Milestone**", and "**Labels**" fields
250  may be left blank.
251 
252 7. Use the "**Submit merge request**" button to create the merge request
253  and visit its page.

255 Review a Merge Request
256 ----------------------
257 
258 Add comments mentioning specific developers using `@username` syntax to
259 draw their attention and have the topic reviewed. After typing `@` and
260 some text, GitLab will offer completions for developers whose real names
261 or user names match.
262 
263 Comments use [GitLab Flavored Markdown][] for formatting. See GitLab
264 documentation on [Special GitLab References][] to add links to things
265 like merge requests and commits in other repositories.
266 
267 When a merge request is ready for review, developers can use the
268 `triage:ready-for-review` to indicate the same to the reviewers. If reviewers
269 deems that it needs more work, they can add the `triage:needswork` label.
270 This can be repeated as many times as needed adding/removing labels as
271 appropriate.
272 
273 If a merge request is waiting on dashboards, use the `triage:pending-dashboards`
274 label.
275

277 [Special GitLab References]: https://gitlab.kitware.com/help/markdown/markdown#special-gitlab-references

279 ### Human Reviews ###
280 
281 Reviewers may add comments providing feedback or to acknowledge their
282 approval. Lines of specific forms will be extracted during
283 [merging](#merge-a-topic) and included as trailing lines of the
284 generated merge commit message:
285 
286 The *leading* line of a comment may optionally be exactly one of the
287 following votes followed by nothing but whitespace before the end
288 of the line:
289 
290 * `-1` or :-1: (`:-1:`) means "The change is not ready for integration."
291 * `+1` or :+1: (`:+1:`) means "I like the change but defer to others."
292 * `+2` means "The change is ready for integration."
293 * `+3` means "I have tested the change and verified it works."
294 
295 The middle lines of a comment may be free-form [GitLab Flavored Markdown][].
296 
297 Zero or more *trailing* lines of a comment may each contain exactly one
298 of the following votes followed by nothing but whitespace before the end
299 of the line:
300 
301 * `Rejected-by: me` means "The change is not ready for integration."
302 * `Acked-by: me` means "I like the change but defer to others."
303 * `Reviewed-by: me` means "The change is ready for integration."
304 * `Tested-by: me` means "I have tested the change and verified it works."
305 
306 Each `me` reference may instead be an `@username` reference or a full
307 `Real Name <user@domain>` reference to credit someone else for performing
308 the review. References to `me` and `@username` will automatically be
309 transformed into a real name and email address according to the user's
310 GitLab account profile.

312 #### Fetching Changes ####
313 
314 One may fetch the changes associated with a merge request by using
315 the `git fetch` command line shown at the top of the Merge Request
316 page. It is of the form:
317 
318  $ git fetch https://gitlab.kitware.com/$username/paraview.git $branch
319 
320 This updates the local `FETCH_HEAD` to refer to the branch.
321 
322 There are a few options for checking out the changes in a work tree:
323 
324 * One may checkout the branch:
325 
326  $ git checkout FETCH_HEAD -b $branch
327  or checkout the commit without creating a local branch:
328 
329  $ git checkout FETCH_HEAD
330 
331 * Or, one may cherry-pick the commits to minimize rebuild time:
332 
333  $ git cherry-pick ..FETCH_HEAD

335 ### Robot Reviews ###
336 
337 The "Kitware Robot" automatically performs basic checks on the commits
338 and adds a comment acknowledging or rejecting the topic. This will be
339 repeated automatically whenever the topic is pushed to your fork again.
340 A re-check may be explicitly requested by adding a comment with a single
341 [*trailing* line](#trailing-lines):
342 
343  Do: check
344 
345 A topic cannot be [merged](#merge-a-topic) until the automatic review
346 succeeds.

348 ### Testing ###
349 
350 ParaView has a [buildbot](http://buildbot.net) instance watching for merge requests
351 to test. A developer must issue a command to buildbot to enable builds:
352 
353  Do: test
354 
355 The buildbot user (@buildbot) will respond with a comment linking to the CDash
356 results when it schedules builds.
357 
358 The `Do: test` command accepts the following arguments:
359 
360  * `--oneshot`
361  only build the *current* hash of the branch; updates will not be built
362  using this command
363  * `--stop`
364  clear the list of commands for the merge request
365  * `--superbuild`
366  build the superbuilds related to the project
367  * `--clear`
368  clear previous commands before adding this command
369  * `--regex-include <arg>` or `-i <arg>`
370  only build on builders matching `<arg>` (a Python regular expression)
371  * `--regex-exclude <arg>` or `-e <arg>`
372  excludes builds on builders matching `<arg>` (a Python regular
373  expression)
374 
375 Multiple `Do: test` commands may be given in separate comments. A new `Do: test`
376 command must be explicitly issued for each branch update for which testing is
377 desired. Buildbot may skip tests for older branch updates that have not started
378 before a test for a new update is requested.
379 
380 Builder names always follow this pattern:
381 
382  project-host-os-libtype-buildtype+feature1+feature2
383 
384  * project: always `paraview` for paraview
385  * host: the buildbot host
386  * os: one of `windows`, `osx`, or `linux`
387  * libtype: `shared` or `static`
388  * buildtype: `release` or `debug`
389  * feature: alphabetical list of features enabled for the build
390 
391 For a list of all builders, see:
392 
393  * [paraview-expected](https://buildbot.kitware.com/builders?category=paraview-expected)
394  * [paraview-superbuild](https://buildbot.kitware.com/builders?category=paraview-superbuild)
395  * [paraview-experimental](https://buildbot.kitware.com/builders?category=paraview-experimental)

397 Revise a Topic
398 --------------
399 
400 If a topic is approved during GitLab review, skip to the
401 [next step](#merge-a-topic). Otherwise, revise the topic
402 and push it back to GitLab for another review as follows:
403 
404 1. Checkout the topic if it is not your current branch:
405 
406  $ git checkout my-topic
407 
408 2. To revise the `3`rd commit back on the topic:
409 
410  $ git rebase -i HEAD~3
411 
412  (Substitute the correct number of commits back, as low as `1`.)
413  Follow Git's interactive instructions.
414 
415 3. Return to the [above step](#share-a-topic) to share the revised topic.
416 
417 Merge a Topic
418 -------------
419 
420 After a topic has been reviewed and approved in a GitLab Merge Request,
421 authorized developers may add a comment of the form
422 
423  Do: merge
424 
425 to ask that the change be merged into the upstream repository. By
426 convention, do not request a merge if any `-1` or `Rejected-by:`
427 review comments have not been resolved and superseded by at least
428 `+1` or `Acked-by:` review comments from the same user.
429 
430 Developers are encouraged to merge their own merge requests on review. However,
431 please do not merge unless you are available to address any dashboard issues that may
432 arise. Developers who repeatedly ignore dashboard issues following their merges may
433 loose developer privileges to the repository temporarily (or permanently)!
434 
435 ### Merge Success ###
436 
437 If the merge succeeds the topic will appear in the upstream repository
438 `master` branch and the Merge Request will be closed automatically.
439 Any issues associated with the Merge Request will generally get closed
440 automatically. If not, the developer merging the changes should close such issues
441 and add a `workflow:customer-review` tag to the issue(s) addressed by the change.
442 Reporters and testers can then review the fix. Try to add enough information to
443 the Issue or the Merge Request to indicate how to test the functionality if not
444 obvious from the original Issue.
445 
446 ### Merge Failure ###
447 
448 If the merge fails (likely due to a conflict), a comment will be added
449 describing the failure. In the case of a conflict, fetch the latest
450 upstream history and rebase on it:
451 
452  $ git fetch origin
453  $ git rebase origin/master
454 
455 (If you are fixing a bug in the latest release then substitute
456 `origin/release` for `origin/master`.)
457 
458 Return to the [above step](#share-a-topic) to share the revised topic.
459 
460 Delete a Topic
461 --------------
462 
463 After a topic has been merged upstream the Merge Request will be closed.
464 Now you may delete your copies of the branch.
465 
466 1. In the GitLab Merge Request page a "**Remove Source Branch**"
467  button will appear. Use it to delete the `my-topic` branch
468  from your fork in GitLab.
469 
470 2. In your work tree checkout and update the `master` branch:
471 
472  $ git checkout master
473  $ git pull
474 
475 3. Delete the local topic branch:
476 
477  $ git branch -d my-topic
478 
479  The `branch -d` command works only when the topic branch has been
480  correctly merged. Use `-D` instead of `-d` to force the deletion
481  of an unmerged topic branch (warning - you could lose commits).

483 Contribute VTK Changes

485

489 
490 1. Create a [ParaView topic](#create-a-topic) if you haven't already.
491 2. Add your VTK topic head (or the latest VTK
492  origin/master which includes your VTK topic head).
493 
494  $ cd VTK
495  $ git checkout master
496  $ cd ..
497  $ git add VTK
498  $ git commit
