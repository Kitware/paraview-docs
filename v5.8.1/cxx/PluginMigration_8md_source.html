1 # Plugin Migration {#PluginMigration}
2 
3 ParaView had a lot of work done to it for the 5.7 release. This involved lots
4 of changes to the CMake Plugin API that ParaView provides to make it easier to
5 build plugins. This document covers the major changes that need to be done in
6 order to migrate to the new API. Specifics of the new CMake API is describled
7 in the [Plugin HowTo](PluginHowto.md) document.
8 
9 ## Metadata
10 
11 Metadata is now stored in a `paraview.plugin` file rather than a `plugin.cmake`
12 file. Instead of being actual CMake code, these files are basically the
13 arguments to a CMake function. Not being CMake code itself allows the new build
14 system to ensure that no extra CMake logic is going on behind its back when
15 it's trying to gather information about the available plugins. Note that this
16 is a default that is supported by the [paraview_plugin_scan][] function. It
17 does not need to be used and the actual files can be named anything as long as
18 the format is valid.
19 
20 The major change is that instead of hiding the `pv_plugin` call with a CMake
21 conditional, one may be written in the `CONDITION` argument which will be used
22 to "hide" the plugin if necessary.
23 
24 Plugins required by the plugin may be listed in the `REQUIRES_MODULES` list so
25 that the module system can ensure that all of them are provided.
26

29 ## Plugins
30 
31 The old `add_paraview_plugin` function has been replaced by
32 [paraview_add_plugin][]. The name change is because the API has been refined.
33 It more consistently uses keyword arguments (e.g., `VERSION 1.0` rather than a
34 hard-coded `1.0` second argument). It is also stricter in its argument parsing
35 so that unrecognized arguments are properly called out. Major changes:
36 
37  - The `DESCRIPTION` has been removed in preference for the `paraview.plugin`
38  entry to remove the duplication it presented.
39  - Due to the new CMake APIs for ClientServer wrapping, filters and classes
40  used in Server Manager XML files (the old `SERVER_MANAGER_SOURCES`
41  argument) must be in a VTK module. These may be added by simply telling the
42  plugin to build some modules using the `MODULE_FILES` and `MODULES`
43  arguments.
44  - `GUI_INTERFACES` is renamed to `UI_INTERFACES` since not all of them are
45  actually graphical in nature.
46  - `GUI_RESOURCES` is renamed to `UI_RESOURCES` for the same reason.
47  - `GUI_SOURCES` is renamed to `SOURCES` since whether they are UI-related or
48  not is immaterial.
49  - `UI_FILES` is provided for Qt's `.ui` interface files.
50  - CMake's AutoGen functionality is used to manage Qt's preprocessing tools
51  (`moc`, `uic`, and `rcc`) rather than the `qt5_wrap_*` functions. It will
52  be enabled for the sources generated by the plugin macro automatically, but
53  any other sources using these should set the associated variables
54  (`CMAKE_AUTOMOC`, `CMAKE_AUTOUIC`, and `CMAKE_AUTORCC`) to ensure it is
55  performed.
56  - `FORCE_STATIC` is provided to force a plugin to be built static or not.
57 
58 [paraview_add_plugin]: @ref paraview_add_plugin

60 ## Interfaces
61 
62 All of the `add_paraview_*` functions which created interfaces for use in the
63 old `add_paraview_plugin(GUI_INTERFACES)` argument have been changed.
64 Generally, names have changed from `add_paraview_X` to `paraview_plugin_add_X`.
65 
66 THe positional arguments for the list of interfaces and sources have been
67 replaced with keyword arguments (`INTERFACES` and `SOURCES` respectively). The
68 new functions also all strictly check their arguments.
69

73  - [paraview_plugin_add_dock_window](@ref paraview_plugin_add_dock_window)
74  - [paraview_plugin_add_action_group](@ref paraview_plugin_add_action_group)

81 ## Modules
82 
83 Modules are now required to have plugin-local classes wrapped for use in Server
84 Manager XML files. This replaces the old `SERVER_MANAGER_SOURCES` argument.
85 Plugin-local modules are installed right beside the plugin library so that they
86 can be moved as a single unit.
87 
88 ## Building Plugins
89 
90 Building plugins now follows a similar pattern as the new VTK module system.
91 Plugins must first be scanned using [paraview_plugin_scan][] to gather all of the
92 metadata and then built using `paraview_plugin_build`.
93 
94 Enable/disable defaults are no longer controlled by the `pv_plugin` function,
95 but instead by the building project. Setting the
96 `_paraview_plugin_default_<PLUGIN_NAME>` variable will select a default enable
97 state for the given plugin.
98 
99 ## Installing Plugins
100 
101 Plugins are installed by [paraview_add_plugin][] using information passed from
102 [paraview_plugin_build][], namely the `INSTALL_EXPORT`, `CMAKE_DESTINATION`,
103 `PLUGINS_COMPONENT`, `TARGET_COMPONENT`, `INSTALL_HEADERS`, the various
104 `_DESTINATION`, and `ADD_INSTALL_RPATHS` arguments. Sets of plugins may be made
105 using [paraview_plugin_build][] now create an interface target (named according
106 to the `TARGET` argument) which may be used by clients to use the plugins in
107 the plugin set.
108 
109 [paraview_plugin_build]: @ref paraview_plugin_build
110 
111 ## Using plugins
112 
113 When building a ParaView client, plugin targets made by
114 [paraview_plugin_build][] may be passed to the [paraview_add_client][]
115 function's `PLUGINS_TARGETS` argument. The client will initialize any static
116 plugins in the set and load plugins marked for auto-loading (via the
117 [paraview_plugin_build][] function's `AUTOLOAD` argument). The plugin target
118 knows where its associated plugin xml file is located and it will be used by
119 the client.
120
