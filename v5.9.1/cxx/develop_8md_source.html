1 Develop ParaView with Git
2 =========================
3 
4 This page documents how to develop ParaView through [Git][].
5 See the [README](README.md) for more information.
6 
7 [Git]: http://git-scm.com
8 
9 Git is an extremely powerful version control tool that supports many
10 different "workflows" for individual development and collaboration.
11 Here we document procedures used by the ParaView development community.
12 In the interest of simplicity and brevity we do *not* provide an
13 explanation of why we use this approach.
14 
15 Setup
16 -----
17 
18 Before you begin, perform initial setup:
19 
20 1. Register [GitLab Access][] to create an account and select a user name.
21 
22 2. [Fork ParaView][] into your user's namespace on GitLab.
23 
24 3. Follow the [download instructions](download.md#clone) to create a
25  local clone of the main ParaView repository. Optionally configure
26  Git to [use SSH instead of HTTPS](download.md#use-ssh-instead-of-https).
27  Then clone:
28 
29  $ git clone --recursive https://gitlab.kitware.com/paraview/paraview.git ParaView
30  $ cd ParaView
31  The main repository will be configured as your `origin` remote.
32 
33 4. Run the [developer setup script][] to prepare your ParaView work tree and
34  create Git command aliases used below:
35 
36  $ ./Utilities/SetupForDevelopment.sh
37  This will prompt for your GitLab user name and configure a remote
38  called `gitlab` to refer to it.
39 
40 5. (Optional but highly recommended.)
41  [Register](https://open.cdash.org/register.php) with the ParaView project
42  on Kitware's CDash instance to better know how your code performs in
43  regression tests. After registering and signing in, click on
44  "All Dashboards" link in the upper left corner, scroll down and click
45  "Subscribe to this project" on the right of ParaView.
46

48 [Fork ParaView]: https://gitlab.kitware.com/paraview/paraview/-/forks/new

51 Workflow
52 --------
53 
54 ParaView development uses a [branchy workflow][] based on topic branches.
55 Our collaboration workflow consists of three main steps:
56 
57 1. Local Development:
58  * [Update](#update)
59  * [Create a Topic](#create-a-topic)
60 
61 2. Code Review (requires [GitLab Access][]):
62  * [Share a Topic](#share-a-topic)
63  * [Create a Merge Request](#create-a-merge-request)
64  * [Review a Merge Request](#review-a-merge-request)
65  * [Revise a Topic](#revise-a-topic)
66 
67 3. Integrate Changes:
68  * [Merge a Topic](#merge-a-topic) (requires permission in GitLab)
69  * [Delete a Topic](#delete-a-topic)
70 
71 [branchy workflow]: http://public.kitware.com/Wiki/Git/Workflow/Topic
72 
73 Update
74 ------
75 
76 1. Update your local `master` branch:
77 
78  $ git checkout master
79  $ git pullall
80 2. Optionally push `master` to your fork in GitLab:
81 
82  $ git push gitlab master
83  to keep it in sync. The `git gitlab-push` script used to
84  [Share a Topic](#share-a-topic) below will also do this.
85 
86 Create a Topic
87 --------------
88 
89 All new work must be committed on topic branches.
90 Name topics like you might name functions: concise but precise.
91 A reader should have a general idea of the feature or fix to be developed given
92 just the branch name. Additionally, it is preferred to have an issue associated with
93 every topic. The issue can document the bug or feature to be developed. In such
94 cases, being your topic name with the issue number.
95 
96 1. To start a new topic branch:
97 
98  $ git fetch origin
99 
100  If there is an issue associated with the topic, assign the issue to yourself
101  using the "**Assignee**" field, and add the
102  `workflow:active-development` label to it.
103 
104 2. For new development, start the topic from `origin/master`:
105 
106  $ git checkout -b my-topic origin/master
107 
108  For release branch fixes, start the topic from `origin/release`, and
109  by convention use a topic name starting in `release-`:
110 
111  $ git checkout -b release-my-topic origin/release
112 
113  If subdmodules may have changed, the run:
114 
115  $ git submodule update
116 
117 3. Edit files and create commits (repeat as needed):
118 
119  $ edit file1 file2 file3
120  $ git add file1 file2 file3
121  $ git commit
122 
123  Caveats:
124  * To add data follow these [vtk instructions][].
125  * To add icons, Kitware's graphic designer may be able to help create an SVG icon.
126 
127  Commit messages must contain a brief description as the first line
128  and a more detailed description of what the commit contains. If
129  the commit contains a new feature, the detailed message must
130  describe the new feature and why it is needed. If the commit
131  contains a bug fix, the detailed message must describe the bug
132  behavior, its underlying cause, and the approach to fix it. If the
133  bug is described in the bug tracker, the commit message must
134  contain a reference to the bug number.
135 
136 4. Add some tests
137 
138  * Start `paraview.exe -dr` to ignore prefs (disable registry)
139  * Choose `Tools .. Record Test` to start.
140  * Choose `Tools .. Lock View Size Custom...` - a 400x400 window works well.
141  * Perform actions in the GUI that exercise your feature. Stop recording.
142  * Put the resulting XML file into `Clients/ParaView/Testing/XML`
143  * Add it to CMakeLists.txt, probably in a TESTS_WITH_BASELINES section
144  * you can manually add `<pqcompareview>` for multiple image comparisons, then add to the TESTS_WITH_INLINE_COMPARES section
145  * Follow the [vtk instructions][] to add the baseline images, which live in `Testing/Data/Baseline/`.
146  * Add your new baseline images to the list in `Testing/XML/CMakeLists.txt`
147  * Add all testing files to your topic.
148 
149  Some background is in the [testing design wiki](https://www.paraview.org/Wiki/Testing_design).
150 
151 5. Add release notes
152 
153  Notable changes should create a new file in `Documentation/release/dev/`
154  named `my-topic.md` (replace `my-topic` with the name of your topic).
155  This is not necessary for branches which are "trivial" such as fixing
156  typos, updating test baselines, or are developer-oriented.
157 
158 [vtk instructions]: https://gitlab.kitware.com/vtk/vtk/-/blob/master/Documentation/dev/git/data.md

160 Share a Topic
161 -------------
162 
163 When a topic is ready for review and possible inclusion, share it by pushing
164 to a fork of your repository in GitLab. Be sure you have registered and
165 signed in for [GitLab Access][] and created your fork by visiting the main
166 [ParaView GitLab][] repository page and using the "Fork" button in the upper right.
167 
168 [ParaView GitLab]: https://gitlab.kitware.com/paraview/paraview
169 
170 1. Checkout the topic if it is not your current branch:
171 
172  $ git checkout my-topic
173 
174 2. Check what commits will be pushed to your fork in GitLab:
175 
176  $ git prepush
177 
178 3. Push commits in your topic branch to your fork in GitLab:
179 
180  $ git gitlab-push
181 
182  Notes:
183  * If you are revising a previously pushed topic and have rewritten the
184  topic history, add `-f` or `--force` to overwrite the destination.
185  * If the topic adds data see [this note](https://gitlab.kitware.com/vtk/vtk/-/blob/master/Documentation/dev/git/data.md#push).
186  * The `gitlab-push` script also pushes the `master` branch to your
187  fork in GitLab to keep it in sync with the upstream `master`.
188 
189  The output will include a link to the topic branch in your fork in GitLab
190  and a link to a page for creating a Merge Request.
191 
192 Create a Merge Request
193 ----------------------
194 
195 (If you already created a merge request for a given topic and have reached
196 this step after revising it, skip to the [next step](#review-a-merge-request).)
197 
198 Visit your fork in GitLab, browse to the "**Merge Requests**" link on the
199 left, and use the "**New Merge Request**" button in the upper right to
200 reach the URL printed at the end of the [previous step](#share-a-topic).
201 It should be of the form:
202 
203  https://gitlab.kitware.com/<username>/paraview/-/merge_requests/new
204 
205 Follow these steps:
206 
207 1. In the "**Source branch**" box select the `<username>/paraview` repository
208  and the `my-topic` branch.
209 
210 2. In the "**Target branch**" box select the `paraview/paraview` repository and
211  the `master` branch. It should be the default.
212 
213  If your change is a fix for the `release` branch, you should still
214  select the `master` branch as the target because the change needs
215  to end up there too.
216 
217 3. Use the "**Compare branches**" button to proceed to the next page
218  and fill out the merge request creation form.
219 
220 4. In the "**Title**" field provide a one-line summary of the entire
221  topic. This will become the title of the Merge Request.
222 
223  Example Merge Request Title:
224 
225  Wrapping: Add Java 1.x support
226 
227 5. In the "**Description**" field provide a high-level description
228  of the change the topic makes and any relevant information about
229  how to try it.
230  * Use `@username` syntax to draw attention of specific developers.
231  This syntax may be used anywhere outside literal text and code
232  blocks. Or, wait until the [next step](#review-a-merge-request)
233  and add comments to draw attention of developers.
234  * If your change is a fix for the `release` branch, indicate this
235  so that a maintainer knows it should be merged to `release`.
236  * Optionally use a fenced code block with type `message` to specify
237  text to be included in the generated merge commit message when the
238  topic is [merged](#merge-a-topic).
239 
240  Example Merge Request Description:
241 
242  This branch requires Java 1.x which is not generally available yet.
243  Get Java 1.x from ... in order to try these changes.
244 
245  ```message
246  Add support for Java 1.x to the wrapping infrastructure.
247  ```
248 
249  Cc: @user1 @user2
250 
251 6. The "**Assign to**", "**Milestone**", and "**Labels**" fields
252  may be left blank.
253 
254 7. Use the "**Submit merge request**" button to create the merge request
255  and visit its page.

257 Review a Merge Request
258 ----------------------
259 
260 Add comments mentioning specific developers using `@username` syntax to
261 draw their attention and have the topic reviewed. After typing `@` and
262 some text, GitLab will offer completions for developers whose real names
263 or user names match.
264 
265 Comments use [GitLab Flavored Markdown][] for formatting. See GitLab
266 documentation on [Special GitLab References][] to add links to things
267 like merge requests and commits in other repositories.
268 
269 When a merge request is ready for review, developers can use the
270 `triage:ready-for-review` to indicate the same to the reviewers. If reviewers
271 deems that it needs more work, they can add the `triage:needswork` label.
272 This can be repeated as many times as needed adding/removing labels as
273 appropriate.
274 
275 If a merge request is waiting on dashboards, use the `triage:pending-dashboards`
276 label.
277

279 [Special GitLab References]: https://gitlab.kitware.com/help/markdown/markdown#special-gitlab-references

281 ### Human Reviews ###
282 
283 Reviewers may add comments providing feedback or to acknowledge their
284 approval. Lines of specific forms will be extracted during
285 [merging](#merge-a-topic) and included as trailing lines of the
286 generated merge commit message:
287 
288 The *leading* line of a comment may optionally be exactly one of the
289 following votes followed by nothing but whitespace before the end
290 of the line:
291 
292 * `-1` or :-1: (`:-1:`) means "The change is not ready for integration."
293 * `+1` or :+1: (`:+1:`) means "I like the change but defer to others."
294 * `+2` means "The change is ready for integration."
295 * `+3` means "I have tested the change and verified it works."
296 
297 The middle lines of a comment may be free-form [GitLab Flavored Markdown][].
298 
299 Zero or more *trailing* lines of a comment may each contain exactly one
300 of the following votes followed by nothing but whitespace before the end
301 of the line:
302 
303 * `Rejected-by: me` means "The change is not ready for integration."
304 * `Acked-by: me` means "I like the change but defer to others."
305 * `Reviewed-by: me` means "The change is ready for integration."
306 * `Tested-by: me` means "I have tested the change and verified it works."
307 
308 Each `me` reference may instead be an `@username` reference or a full
309 `Real Name <user@domain>` reference to credit someone else for performing
310 the review. References to `me` and `@username` will automatically be
311 transformed into a real name and email address according to the user's
312 GitLab account profile.
313 
314 #### Fetching Changes ####
315 
316 One may fetch the changes associated with a merge request by using
317 the `git fetch` command line shown at the top of the Merge Request
318 page. It is of the form:
319 
320  $ git fetch https://gitlab.kitware.com/$username/paraview.git $branch
321 
322 This updates the local `FETCH_HEAD` to refer to the branch.
323 
324 There are a few options for checking out the changes in a work tree:
325 
326 * One may checkout the branch:
327 
328  $ git checkout FETCH_HEAD -b $branch
329  or checkout the commit without creating a local branch:
330 
331  $ git checkout FETCH_HEAD
332 
333 * Or, one may cherry-pick the commits to minimize rebuild time:
334 
335  $ git cherry-pick ..FETCH_HEAD

337 ### Robot Reviews ###
338 
339 The "Kitware Robot" automatically performs basic checks on the commits
340 and adds a comment acknowledging or rejecting the topic. This will be
341 repeated automatically whenever the topic is pushed to your fork again.
342 A re-check may be explicitly requested by adding a comment with a single
343 [*trailing* line](#trailing-lines):
344 
345  Do: check
346 
347 A topic cannot be [merged](#merge-a-topic) until the automatic review
348 succeeds.

350 ### Testing ###
351 
352 ParaView uses [GitLab CI][] to test merge requests, configured by the top-level
353 `.gitlab-ci.yml` file. Results may be seen both on the merge request's
354 pipeline page and on the [ParaView CDash Page][]. Filtered CDash results
355 showing just the pipeline's jobs can be reached by selecting the `cdash-commit`
356 job in the `External` stage of the pipeline. Note that due to GitLab changes,
357 the `External` stage may be in a separate pipeline for the same commit.
358 
359 Lint build jobs run automatically after every push. Heavier jobs require a
360 manual trigger to run:
361 
362 * Merge request authors may visit their merge request's pipeline and click the
363  "Play" button on one or more jobs manually. If the merge request has the
364  "Allow commits from members who can merge to the target branch" check box
365  enabled, ParaView maintainers may use the "Play" button too.
366 
367 * [ParaView GitLab Project Developers][] may trigger CI on a merge request by
368  adding a comment with a command among the [comment trailing
369  lines](#trailing-lines):
370 
371  Do: test
372 
373  `@kwrobot` will add an award emoji to the comment to indicate that it
374  was processed and also trigger all manual jobs in the merge request's
375  pipeline.
376 
377  The `Do: test` command accepts the following arguments:
378 
379  * `--named <regex>`, `-n <regex>`: Trigger jobs matching `<regex>` anywhere
380  in their name. Job names may be seen on the merge request's pipeline page.
381  * `--stage <stage>`, `-s <stage>`: Only affect jobs in a given stage. Stage
382  names may be seen on the merge request's pipeline page. Note that the
383  names are determined by what is in the `.gitlab-ci.yml` file and may be
384  capitalized in the web page, so lowercasing the webpage's display name for
385  stages may be required.
386  * `--action <action>`, `-a <action>`: The action to perform on the jobs.
387  Possible actions:
388 
389  * `manual` (the default): Start jobs awaiting manual interaction.
390  * `unsuccessful`: Start or restart jobs which have not completed
391  successfully.
392  * `failed`: Restart jobs which have completed, but without success.
393  * `completed`: Restart all completed jobs.
394 
395 If the merge request topic branch is updated by a push, a new manual trigger
396 using one of the above methods is needed to start CI again. Currently running
397 jobs will generally be canceled automatically.

403 Revise a Topic
404 --------------
405 
406 If a topic is approved during GitLab review, skip to the
407 [next step](#merge-a-topic). Otherwise, revise the topic
408 and push it back to GitLab for another review as follows:
409 
410 1. Checkout the topic if it is not your current branch:
411 
412  $ git checkout my-topic
413 
414 2. To revise the `3`rd commit back on the topic:
415 
416  $ git rebase -i HEAD~3
417 
418  (Substitute the correct number of commits back, as low as `1`.)
419  Follow Git's interactive instructions.
420 
421 3. Return to the [above step](#share-a-topic) to share the revised topic.
422 
423 Merge a Topic
424 -------------
425 
426 After a topic has been reviewed and approved in a GitLab Merge Request,
427 authorized developers may add a comment of the form
428 
429  Do: merge
430 
431 to ask that the change be merged into the upstream repository. By
432 convention, do not request a merge if any `-1` or `Rejected-by:`
433 review comments have not been resolved and superseded by at least
434 `+1` or `Acked-by:` review comments from the same user.
435 
436 Developers are encouraged to merge their own merge requests on review. However,
437 please do not merge unless you are available to address any dashboard issues that may
438 arise. Developers who repeatedly ignore dashboard issues following their merges may
439 loose developer privileges to the repository temporarily (or permanently)!
440 
441 ### Merge Success ###
442 
443 If the merge succeeds the topic will appear in the upstream repository
444 `master` branch and the Merge Request will be closed automatically.
445 Any issues associated with the Merge Request will generally get closed
446 automatically. If not, the developer merging the changes should close such issues
447 and add a `workflow:customer-review` tag to the issue(s) addressed by the change.
448 Reporters and testers can then review the fix. Try to add enough information to
449 the Issue or the Merge Request to indicate how to test the functionality if not
450 obvious from the original Issue.
451 
452 ### Merge Failure ###
453 
454 If the merge fails (likely due to a conflict), a comment will be added
455 describing the failure. In the case of a conflict, fetch the latest
456 upstream history and rebase on it:
457 
458  $ git fetch origin
459  $ git rebase origin/master
460 
461 (If you are fixing a bug in the latest release then substitute
462 `origin/release` for `origin/master`.)
463 
464 Return to the [above step](#share-a-topic) to share the revised topic.
465 
466 Delete a Topic
467 --------------
468 
469 After a topic has been merged upstream the Merge Request will be closed.
470 Now you may delete your copies of the branch.
471 
472 1. In the GitLab Merge Request page a "**Remove Source Branch**"
473  button will appear. Use it to delete the `my-topic` branch
474  from your fork in GitLab.
475 
476 2. In your work tree checkout and update the `master` branch:
477 
478  $ git checkout master
479  $ git pull
480 
481 3. Delete the local topic branch:
482 
483  $ git branch -d my-topic
484 
485  The `branch -d` command works only when the topic branch has been
486  correctly merged. Use `-D` instead of `-d` to force the deletion
487  of an unmerged topic branch (warning - you could lose commits).

489 Contribute VTK Changes
490 ----------------------
491 
492 If you have any VTK changes, then you are required to get your changes
493 incorporated into VTK using [VTK's development workflow][]. Once your VTK topic has
494 been approved and merged into VTK, then:
495 
496 1. Create a [ParaView topic](#create-a-topic) if you haven't already.
497 2. Add your VTK topic head (or the latest VTK
498  origin/master which includes your VTK topic head).
499 
500  $ cd VTK
501  $ git checkout master
502  $ cd ..
503  $ git add VTK
504  $ git commit
505 
506 3. Follow the merge process documented earlier.
