1 # Anatomy of Catalyst Python Module (Version 2.0) {#CatalystPythonScriptsV2}
2 
3 This page describes the details of Python modules intended for use in Catalyst.
4 This page covers features supported by ParaView 5.9 and later i.e. version 2.0.
5 
6 ## Basics
7 
8 Catalyst Python modules are simply Python scripts that use the `paraview`
9 package to define data analysis and visualization pipelines. You use these
10 modules to define the analysis pipelines to be executed in a
11 Catalyst-instrumented simulation code, for example.
12 
13 [Python](https://docs.python.org/3/tutorial/modules.html)
14 defines a module as a file containing Python definitions and statements.
15 The file name for a module is the name of the module suffixed by extension
16 `.py`. Thus a module named `foo` will be in a file named `foo.py`.
17 
18 [Packages](https://docs.python.org/3/tutorial/modules.html#packages) are a
19 mechanism for structuring multiple modules. Essentially, if you place a bunch of
20 .py files (aka modules) in a directory structure, that becomes a package! Of
21 course, there's a little more to that and the
22 [Python docs](https://docs.python.org/3/tutorial/modules.html#packages) are
23 a highly recommended read. Similar to the convection to name a module using the
24 file name, the name of the directory representing a Python package
25 becomes the name of the package. A package can contain multiple directories
26 (sub-packages) and Python files.
27 
28 When a Python module is *imported* into the application, the Python statements
29 in the module are executed. Since a Python module is a file, the execution
30 starts with the first statement in the file and continues till the end of the
31 file is reached or an error is encountered.
32 
33 When a Python package is *imported*, the entry-point is defined by a file named
34 `__init__.py` with in the package. Statements in this file are executed. In the
35 simplest case, `__init__.py` can be an empty file, but it can also execute
36 initialization code, including importing other modules, packages, sub-packages
37 etc.
38 
39 Catalyst Python code can be structured as modules or packages. In either case,
40 the module (or package) is imported and then functions or variables defined on
41 the module (or package) are checked and invoked as described in following
42 sections.
43 
44 ## vtkCPPythonScriptV2Pipeline
45 
46 Catalyst adaptors should use `vtkCPPythonScriptV2Pipeline` to execute scripts as
47 described in this document. For older Catalyst Python scripts, i.e. those that
48 use `RequestDataDescription` and `DoCoProcess` functions, should continue to
49 use `vtkCPPythonScriptPipeline` and `vtkCPPythonStringPipeline` at this time.
50 It is conceivable that we provide a wrapper that lets us use
51 `vtkCPPythonScriptV2Pipeline` for these older script too, however, that has not
52 been implemented at this time.

55 
56 Now, let's look at the structure of a Catalyst Python module. The same is
57 applicable to a Python package as well.

59 ### Visualization Pipeline
60 
61 ParaView Python API, exposed via the `paraview.simple` module, is used to define
62 the visualization and data analysis pipeline. The API mimics actions one would
63 take the GUI. For example, simple script to create a **Wavelet** source and
64 slice it, looks as follows:
65 
66 ```py
67 # filename: sample1.py
68 from paraview.simple import *
69 from paraview import print_info
70 
71 # print start marker
72 print_info("begin '%s'", __name__)
73 
74 wavelet1 = Wavelet(registrationName='Wavelet1')
75 
76 # create a new 'Slice'
77 slice1 = Slice(registrationName='Slice1', Input=wavelet1)
78 slice1.SliceType = 'Plane'
79 slice1.SliceType.Normal = [0, 0, 1]
80 
81 # print end marker
82 print_info("end '%s'", __name__)
83 ```
84 
85 This same code can be used in a Catalyst Python script to setup the visualization
86 pipeline. The only thing to note is that we need a mechanism to indicate which of
87 the data-producers in the pipeline should be replaced by the data generated by the
88 simulation. In Catalyst, the data produced by the simulation is available on named
89 channels. Thus, we need to identify which named-channel corresponds to which data
90 data-producer in the pipeline. For that, we use the `registrationName` attribute.
91 In the Python script, when creating a source or a filter, one can pass in an
92 optional argument `registrationName`. When the script is being executed within
93 Catalyst, Catalyst looks to see if the `registrationName` matches the name of a
94 known channel. If so, the data producer type requested (in this case Wavelet) is
95 ignored and instead is replaced by a producer that provides simulation data as
96 the output.
97 
98 The `paraview.demos.wavelet_miniapp` is a miniapp that acts as a simulation
99 producing a time value `vtkImageData` dataset. So we can use it to run the above
100 script as follows:

118 Let's inspect the warnings to understand what's going on.
119 
120 `bridge.py:17` warning can be ignored for now. The `INFO` messages are produced
121 by `wavelet_miniapp.py` to provide information about the timestep as it
122 progresses through the mock-simulation timesteps.
123 
124 `Module 'sample1' missing Catalyst 'options', will use a default options object`
125 warning lets us know that the Catalyst will default 'options'. More on that
126 later.
127 
128 `script may not depend on simulation data; is that expected?` tells us that the
129 pipeline we've setup in this script does not use any data produced by the
130 simulation at all. While there's nothing wrong with that, it's highly unlikely
131 that that's the intent and hence this warning is posted.
132 
133 In our case, we want the `Wavelet` source to be replaced by the simulation data.
134 As mentioned, this means the `registrationName` should match the simulation
135 channel name. `wavelet_miniapp` uses **input** as the default channel name. It
136 can be changed using `-c` (or `--channel`) command line argument. Let's
137 rename the channel using `--channel`.
138 
139 ```
140 > pvbatch -m paraview.demos.wavelet_miniapp
141  --script .../sample1.py
142  --script-version 2
143  --timesteps 2
144  --channel Wavelet1
145 
146 ( 0.563s) [pvbatch ] bridge.py:17 WARN| Warning: ParaView has been initialized before `initialize` is called
147 ( 1.876s) [pvbatch ] wavelet_miniapp.py:97 INFO| timestep: 1/2
148 ( 1.944s) [pvbatch ] sample1.py:6 INFO| begin 'sample1'
149 ( 2.069s) [pvbatch ] sample1.py:16 INFO| end 'sample1'
150 ( 2.069s) [pvbatch ] v2_internals.py:145 WARN| Module 'sample1' missing Catalyst 'options', will use a default options object
151 ( 2.072s) [pvbatch ] wavelet_miniapp.py:97 INFO| timestep: 2/2

154 
155 Great! Now the warnings about simulation data have disappeared.
156 
157 The pipeline, however, still doesn't do anything significant.
158 
159 Also, the code in the module seems to execute only once, for the first timestep.
160 This is confirmed by the fact that the `print_info` statements in `sample.py` only show
161 up once in the output. This is because the code in the module gets executed
162 only when its imported for the first time.
163 Once imported, which happens on first timestep, the module is not re-imported and
164 hence the code is not re-executed.
165 
166 To add code that gets executed each iteration, you can define a
167 `catalyst_execute` function. Likewise, `catalyst_initialize` and
168 `catalyst_finalize` functions may be defined too and they will get called for
169 the first and after the last timestep respectively. Here's an example script
170 to see that in action.
171 
172 ```py
173 # filename: sample2.py
174 from paraview.simple import *
175 from paraview import print_info
176 
177 # print start marker
178 print_info("begin '%s'", __name__)
179 
180 w = Wavelet(registrationName="Wavelet1")
181 
182 def catalyst_initialize():
183  print_info("in '%s::catalyst_initialize'", __name__)
184 
185 def catalyst_execute(info):
186  print_info("in '%s::catalyst_execute'", __name__)
187 
188 def catalyst_finalize():
189  print_info("in '%s::catalyst_finalize'", __name__)
190 
191 # print end marker
192 print_info("end '%s'", __name__)
193 ```

306 ### Using Extractors
307 
308 In ParaView 5.9, we introduced a new pipeline object called **Extractors**.
309 **Extractors** are items in the visualization pipeline that can save data or
310 images per timestep. Extractors make it largely unnecessary to have any custom code
311 to execute per iteration since one can simply use extracts
312 to save out image extracts from views or data extracts from filters and other
313 data producers.
314 
315 
316 For example, `sample3.py` can be modified to use extractors as follows:
317 
318 ```py
319 # filename: sample4.py
320 from paraview.simple import *
321 from paraview import print_info
322 
323 view = CreateRenderView()
324 
325 wavelet1 = Wavelet(registrationName="Wavelet1")
326 
327 # create a new 'Slice'
328 slice1 = Slice(registrationName='Slice1', Input=wavelet1)
329 slice1.SliceType = 'Plane'
330 slice1.SliceType.Normal = [0, 0, 1]
331 
332 sliceDisplay = Show(slice1)
333 
334 # create extractor
335 extractor1 = CreateExtractor('PNG', view, registrationName='PNG1')
336 # trace defaults for the extractor.
337 # init the 'PNG' selected for 'Writer'
338 extractor1.Writer.FileName = '/tmp/output-%ts.png'
339 extractor1.Writer.ImageResolution = [800, 880]
340 ```
341 
342 This will produce an output like follows:
343 ```
344 > pvbatch -m paraview.demos.wavelet_miniapp
345  -s .../sample3.py
346  --script-version 2
347  --timesteps 3
348  -c Wavelet1
349 
350 ( 0.554s) [pvbatch ] bridge.py:17 WARN| Warning: ParaView has been initialized before `initialize` is called
351 ( 1.835s) [pvbatch ] wavelet_miniapp.py:97 INFO| timestep: 1/3
352 ( 2.823s) [pvbatch ] v2_internals.py:150 WARN| Module 'sample4' missing Catalyst 'options', will use a default options object
353 ( 3.208s) [pvbatch ] wavelet_miniapp.py:97 INFO| timestep: 2/3
354 ( 3.652s) [pvbatch ] wavelet_miniapp.py:97 INFO| timestep: 3/3
355 ```
356 
357 If you look at the generated results, `/tmp/output-0.png`, `/tmp/output-1.png`,
358 etc. will be generated as with `sample3.py`. However, this time, we didn't need
359 to add an explicit code to execute per iteration by providing `catalyst_execute`.
360 The extractor achieve the same effect.
361 
362 Such Catalyst scripts that use extractors can be easily exported from the
363 ParaView GUI using **File > Save Catalyst State**. Such an exported script will
364 often how the following trailer:
365 
366 ```py
367 # exported Catalyst script
368 ...
369 
370 # ------------------------------------------------------------------------------
371 # Catalyst options
372 from paraview import catalyst
373 options = catalyst.Options()
374 options.GenerateCinemaSpecification = 1
375 options.ExtractsOutputDirectory = "/tmp"
376 
377 # ------------------------------------------------------------------------------
378 if __name__ == '__main__':
379  from paraview.simple import SaveExtractsUsingCatalystOptions
380  # Code for non in-situ environments; if executing in post-processing
381  # i.e. non-Catalyst mode, lets generate extracts using Catalyst options
382  SaveExtractsUsingCatalystOptions(options)
383 ```
384 
385 The `options` variable is setup with values chosen in the **Save Catalyst State** dialog.
386 This is the *options* variable that we have been seeing warnings for in our
387 runs so far. If present, `options` must be of the type
388 `paraview.catalyst.Options`. This is used to specify parameters that control how
389 the code is executed e.g. location of output directory, Catalyst Live connection
390 information etc. If not present, a default one is created.
391 
392 The following lists several of the important properties on `options` and their
393 default values.
394 
395 ```py
396 # catalyst options
397 from paraview.catalyst import Options
398 options = Options()
399 
400 # directory under which to save all extracts
401 # generated using Extractors defined in the pipeline, if any.
402 # (optional, but recommended)
403 options.ExtractsOutputDirectory = "...."
404 
405 # enable/disable Cinema database generation for
406 # generated extracts (optional)
407 options.GenerateCinemaSpecification = ... # default=False
408 
409 # global trigger params (optional)
410 options.GlobalTrigger.UseStartTimeStep = ... # default=False
411 options.GlobalTrigger.StartTimeStep = ... # default=0
412 options.GlobalTrigger.UseEndTimeStep = .. # default=False
413 options.GlobalTrigger.EndTimeStep = ... # default=0
414 options.GlobalTrigger.Frequency = ... # default=1
415 
416 # live params (optional)
417 options.EnableCatalystLive = ... # default=False
418 options.CatalystLiveURL = ... # default="localhost:2222"
419 options.CatalystLiveTrigger.UseStartTimeStep = ... # default=False
420 options.CatalystLiveTrigger.StartTimeStep = ... # default=0
421 options.CatalystLiveTrigger.UseEndTimeStep = ... # default=False
422 options.CatalystLiveTrigger.EndTimeStep = ... # default=0
423 options.CatalystLiveTrigger.Frequency = ... # default=1
424 
425 ```
426 
427 `options.ExtractsOutputDirectory` is handy to place all generated extracts under
428 a specific directory. To use it, ensure that all extractor use a relative
429 filename. The filename is then evaluated to be relative to the directory
430 provided for `options.ExtractsOutputDirectory`

476 ### Using ZIP archives
477

479 having to unzip the contents. This can be a very handy feature for Catalyst
480 Python scripts. You can simply archive a package directory using the same name

507 # Testing and Debugging
508 
509 The `pvpython`/`pvbatch` executables can be launched with a `-l` argument to
510 generate detailed logs as follows:
511 
512 ```bash
513 > pvbatch -l=/tmp/log.txt,TRACE -m paraview.demos.wavelet_miniapp ...
514 ```
515 
516 Here, a `/tmp/log.txt` will all logging output will be generated. You can
517 also elevate Catalyst-generated log to a higher level and log that, for example,
518 the following will elevate catalyst log level to INFO and log all INFO messages
519 to `/tmp/infolog.txt`
520 
521 ```bash
522 > env PARAVIEW_LOG_CATALYST_VERBOSITY=INFO \
523  pvbatch -l=/tmp/infolog.txt,INFO -m paraview.demos.wavelet_miniapp ...
524 ```

527 ## Mini-App: paraview.demos.wavelet\_miniapp
528 
529 The `wavelet_miniapp` that we used in several of the examples here is a simple
530 stand-in for a simulation code. It uses the Wavelet (`vtkRTAnalyticSource`) internally
531 to produce time-varying uniform rectilinear grid (`vtkImageData`). Command line
532 options can be used to customize the run, including number of timesteps, etc.
533 
534 ```
535 > ./bin/pvbatch -m paraview.demos.wavelet_miniapp --help
536 
537 usage: wavelet_miniapp.py [-h] [-t TIMESTEPS] [--size SIZE] -s SCRIPT [--script-version SCRIPT_VERSION] [-d DELAY] [-c CHANNEL]
538 
539 Wavelet MiniApp for Catalyst testing
540 
541 optional arguments:
542  -h, --help show this help message and exit
543  -t TIMESTEPS, --timesteps TIMESTEPS
544  number of timesteps to run the miniapp for (default: 100)
545  --size SIZE number of samples in each coordinate direction (default: 101)
546  -s SCRIPT, --script SCRIPT
547  path(s) to the Catalyst script(s) to use for in situ processing.
548  Can be a .py file or a Python package zip or directory
