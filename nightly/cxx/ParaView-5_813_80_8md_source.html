20 ## GLTF importer
21 
22 You can now import all or a subset of meshes from a GLTF 2.0 file into ParaView as well as the material properties supported in VTK and textures.

34 ## New fullscreen mode for active view
35 
36 The previously supported fullscreen mode sets the tabbed multi-view widget fullscreen, that is, all views are maximized, including separators and decoration.
37 
38 ParaView now has a new fullscreen mode showing only the current active view in fullscreen (shortcut: Ctrl + F11). The new mode is intended to be closer to a "classical" fullscreen mode.
39 
40 This is in part motivated by the need of some hardware (like the zSpace Inspire) to have a strict fullscreen mode in order to display the rendered scene correctly.
41 
42 ## Select Blocks Through
43 
44 ParaView's RenderView now has a new selection action called "Select Blocks Through".
45 This action allows you to select all blocks that are inside the selected frustum.
46 This is useful for selecting blocks that are hidden behind other blocks.
47 
48 ## _MultiBlock Inspector_ support for coloring blocks by different arrays
49 
50 In the previous version of ParaView, _MultiBlock Inspector_ supported changing only the following block properties:
51 
52 * **Visibility**
53 * **Color**
54 * **Opacity**
55 
56 All properties were previously set through controls in the multiblock tree.
57 
58 Now, the _MultiBlock Inspector_ supports changing the following the block properties:
59 
60 * **Visibility** via a column in the multiblock tree
61 * **Color**
62 * **Color Array**
63 * **Use Separate Color Map**
64 * **Interpolate Scalars Before Mapping**
65 * **Opacity**
66 * **Map Scalars**
67 
68 With the exception of the **Visibility** property, all the properties in the new inspector are set in a small block color property panel below the multiblock tree. The **Visibility** property is set via a checkbox in the multiblock tree.
69 
70 Each property has a state indicator and can be reset to the default value. All block selectors also have a state and a reset button for all of the properties of a block or parent of blocks.
71 
72 To change a property of a block/blocks, the select the blocks in the _MultiBlock Inspector_ and then change the desired property in the block color property panel. The change will be applied to all selected blocks.
73 
74 > ![New _MultiBlock Inspector_](img/5.13.0/MultiBlockInspectorNew.png)
75 >
76 > The revised _MultiBlock Inspector_ enables setting additional per-block properties by controls in the block color property panel.

78 ## VTKHDF file writer
79 
80 A VTKHDF writer and extractor are now available in ParaView.
81 
82 Currently, this writer supports writing Unstructured Grid and Polygonal Meshes, with time series support, as well as composite types Partitioned Dataset Collection and Multi-block Dataset, without time series support. This writer can also write compressed data to save storage space.
83 
84 The VTKHDF exporter writes time-dependent data as file series, not using the time series support in the VTKHDF writer.

86 ## Command-line option to choose `pvserver` socket bind address
87 
88 Add the `--bind-address` command-line option to `pvserver`. This option allows changing the address bound to the server socket, so that it is not necessarily exposed on all interfaces using `INADDR_ANY` (0.0.0.0).
89 
90 ## Server-only command-line options configuration file
91 
92 Settings are now supported for `pvserver`, `pvrenderserver` and `pvdataserver`.
93 
94 These executables parse a file named `ParaViewServer-UserSettings.json` and look for settings named according to `pvserver` command-line connection options.
95 
96 This file is expected to look like this:
97 
98 ```
99 {
100  "cli-options" :
101  {
102  "connection" :
103  {
104  "connect-id" : 17,
105  "reverse-connection" : true,
106  "client-host" : "bailey"
107  }
108  }
109 }
110 ```
111 
112 The name of the setting is the "long" version of the option, without the two leading dashes.
113 
114 **only connection options are supported.**

117 # Rendering enhancements
118 
119 ## Geometry representations can compute surface normals
120 
121 Geometry representations can now compute surface cell and point normals by enabling the new **Compute Point Normals** property. This can be a convenience to avoid the need to add an **Extract Surface** filter followed by a **Generate Surface Normals** filter. This option is available for the **Surface**, **Surface with Edges**, and the **Surface LIC** representations.
122 
123 ## Faster time to first render with static meshes
124 
125 ParaView now uses a cache to save time when computing surfaces to display. If the surface mesh is unchanged since the last time it was extracted and caching is enabled, the extraction step simply uses the cached result, saving significant work in the steps leading up to the first render after a change is made.
126 
127 This feature is not available in distributed execution.
128 
129 ## New **Polar Grid** annotation in **RenderView**s
130 
131 **RenderView**s now have a new annotation available: the **Polar Grid**.
132 
133 Similar to the **Polar Axes** already existing the Representation level, the **Polar Grid** can display axes and arcs, annotated with polar coordinates.
134 
135 Like the **Legend Grid**, this new annotation is drawn in the screen space. Thus, it is only available when **Camera Parallel Projection** is enabled.
136 
137 > ![Polar Grid Annotation](img/5.13.0/polar-grid-annotation.png)
138 >
139 > Example of a **Polar Grid** annotation
140 
141 ## Python block coloring functionality
142 
143 ParaView now has new block coloring capabilities thanks to additional block-specific properties and Python functions that apply to representations. New Python functions including `ColorBlocksBy()` and `GetBlockColorTransferFunction()` are also available to script block coloring options. An example can be found
144 in `Clients/ParaView/Testing/Python/ColorBlockBy.py` test in ParaView's source code.
145 
146 ## Surface representation **Texture Transform** property
147 
148 The new **Texture Transform** property has been added to translate, rotate, and scale texture coordinates when a texture is applied to an object. Since this transform can flip a texture by using a negative scale factor, the **Flip Textures** property has been deprecated.
149 
150 ## Configurable maximum radius for **Polar Axes**
151 
152 You can now configure a custom maximum radius for the **Polar Axes**. Note that by default the custom maximum radius is off and therefore it is automatically computed relative to the pole position.
153 
154 ## Constant shading per object feature
155 
156 Before this feature, you could disable the lighting only for a whole view using the **Light Kit** checkbox inside the **Light Inspector**. This change allows you to disable lighting for a single source with a new display property named **Disable Lighting**.

161 ## zSpace Inspire support in ParaView
162 
163 The zSpace plugin now uses a dedicated render window class to support the specific rendering mode of the latest zSpace hardware. Unlike previous models, the zSpace Inspire does not rely on quad buffering to display stereo content. It uses its dedicated stereo display instead to perform stereo rendering. In order to have a working stereo effect, the current active view should be shown in fullscreen. Note that the zSpace Stereo Core Compatibility API should be used with this display.

166 # Filter changes
167 
168 ## **Axis-Aligned Slice** filter
169 
170 The **Axis-Aligned Slice** filter is now available in ParaView.
171 
172 This filter produces axis-aligned slices of the input data. The output data type is the same as the input, effectively reducing the dimensionality of the input data by one.
173 
174 For now, this filter supports Hyper-tree Grids and Overlapping AMR Datasets as input data, and **Axis-Aligned plane** as cut function. In the case of Hyper-tree Grids, it also supports producing multiple slices at once.
175 
176 This filter also supports composite datasets of Hyper-tree Grids. In such cases, it will iterate over each Hyper-tree Grid and generate a new hierarchy with added nodes containing the slices.
177 
178 Please note that usages of **Axis-Aligned plane** in the **Slice** and **Slice With Plane** filters are deprecated in favor of this new dedicated filter.
179 
180 > ![Results showing multiple axis-aligned slices from a Hyper-tree Grid.](img/5.13.0/htg-slice-axis-aligned.png)
181 >
182 > Results showing multiple axis-aligned slices from a Hyper-tree Grid.
183 
184 ## **Force Static Mesh** filter
185 
186 Introduce a new filter, named **Force Static Mesh**, that explicitly advertises that a mesh has not been modified. This information can be exploited by subsequent filters like the **Surface Filter** for performance and memory optimizations.
187 
188 ## **Feature Edges** filter available for Hyper-tree Grids
189 
190 You can now apply the **Feature Edges** filter in ParaView to Hyper-tree Grids. The **Feature Edges** representation now works correctly with Hyper-tree Grids as well.
191 
192 > ![Filter Edges filter results applied to a Hyper-tree Grid](img/5.13.0/htg-features-edges.png)
193 >
194 > **Feature Edges** filter available for Hyper-tree Grids.
195 
196 ## New **Compute Visible Leaves Size** filter for Hyper-tree Grids
197 
198 A new Hyper-tree Grid utility filter is available: **HyperTreeGrid Visible Leaves Size**. This filter creates two new cell fields:
199 - `ValidCell` has a (double) value of 1.0 for visible (not ghost, non masked) leaf cells, and 0.0 for the others.
200 - `CellSize`'s value corresponds to the volume of the cell in 3D, or area in 2D.
201 
202 ## Add Hyper-tree Grid support to various data generation filters
203 
204 Following data generation filters now support Hyper-tree Grids as input:
205 
206 - **Point And Cell Ids**
207 - **Global Point And Cell Ids**
208 - **Process Ids**
209 - **Random Attributes**
210 
211 Note that only cell data can be generated for Hyper-tree Grids.
212 
213 ## Mask generation for the **Hyper Tree Grid (Random)** source
214 
215 You can now generate a mask for the **Hyper Tree Grid (Random)** source, it can be controlled by the property **Masked Fraction** that controls the spatial proportion of the Hyper-tree Grid that will be masked. This value is a target and the actual masked fraction of the generated Hyper-tree Grid can differ up to an error margin, depending on the number of trees and the branching factor. Which means a masked fraction with a value near 1 can lead to a not completely masked Hyper-tree Grid, since the error margin still applies for this value.
216 
217 > ![Example of mask generation in the Hyper Tree Grid (Random) source](img/5.13.0/htg-source-random.png)
218 >
219 > Mask generation for the **Hyper Tree Grid (Random)** source
220 
221 ## Multi-slice support in **Slice** filter for Hyper-tree Grids
222 
223 The **Slice** filter can now generate mutliple slices at once when the input is a Hyper-tree Grid.
224 
225 Please note that this feature is not available when the **Axis Aligned Plane** is selected as the cut function, since this option is deprecated in the **Slice** filter.
226 You can now use the dedicated **Axis-Aligned Slice** filter to do so.
227 
228 ## More ghost-related filters are available for Hyper-tree Grids
229 
230 The **Extract Ghost Cells** and **Remove Ghost Cells** filters can now be used for Hyper-tree Grids. Note that these filters mask cells and do not impact the number of cells of the input Hyper-tree Grids.
231 
232 ## New **Critical Time** filter
233 
234 Given an input that changes over time, the `Critical Time` filter generates a new data array containing time step values. These values correspond to the time at which a specified threshold criterion has been met for a given point/cell array (at each point/cell respectively).
235 
236 To do so, the filter needs to process all available timesteps. The output of the filter is not temporal.
237 
238 Like in the `Threshold` filter, the threshold criterion can take three forms:
239 
240 * greater than a particular value
241 * less than a particular value
242 * between two values
243 
244 # New **Temporal Smoothing** filter
245 
246 ParaView now has a **Temporal Smoothing** filter that smooths point and cell data from a temporal source by averaging the values over a temporal window defined by the filter's **Temporal Window Half Width**.
247 
248 
249 ## Angle-based and custom field strategies for polyline decimation
250 
251 The **Decimate Polyline** filter now support multiple decimation strategies through a **Decimation Strategy** property:
252 
253 - `DecimationAngleStrategy`: use the angle between three consecutive points as error metric
254 - `DecimationDistanceStrategy` (default, identical to previous versions): use the distance between consecutive points as error metric
255 - `DecimationCustomFieldStrategy`: use values stored in a custom PointData array to compute the error between three consecutive points. The name of this array must be defined using the **Field Name** property.
256

258 # Changes in readers and writers
259 
260 ## Add ERF Reader
261 
262 You can now read ERF HDF5 files into ParaView. The reader supports version 1.2 of the ERF [HDF5 spec](https://myesi.esi-group.com/ERF-HDF5/doc/ERF_HDF5_Specs_1.2.pdf), specifically:
263 
264 * Reading a selected stage.
265 * Reading the 'constant' group.
266 * Reading the 'singlestate' group.
267 
268 The output of the reader is a Partitioned Dataset Collection composed of multiple Unstructured Grids for the 'constant' or 'singlestate' groups.
269 
270 ## IOSS reader caching option
271 
272 The IOSS Reader now has a **Caching** property. When enabled, the reader will cache the mesh data in memory across multiple time steps. Previously, when the option was not present, the reader would always cache data, which could lead to excessive memory usage. Now, the default behavior is to not cache.
273 
274 ## Fixing data layout issues for the **openPMD reader**
275 
276 The openPMD format allows storage of multidimensional arrays (meshes) in either `Fortran` or `C` data orders with arbitrary axis labeling. Based on the data order, it may be necessary to transpose the data so that ParaView can correctly interpret the meshes.
277 
278 * For `C` data order, the VTK axis ordering is expected to be (x, y, z)
279 * For `F` data order, the VTK axis ordering is expected to be (z, y, x)
280 
281 Transposes are necessary is there is a mismatch in the expected axis ordering. This also fixes problems with 1D/2D meshes that require handling the transposes differently than 3D meshes.
282 
283 ## Option in the GLTFExporter to avoid saving NaN color in exported color map textures
284 
285 A new property has been added to the GLTF Exporter. **Save NaN Value** controls whether the exporter should save the NaN color in the exported texture. If enabled, the NaN color is saved as a second row in the color map texture which might affect how the texture is processed by the application that loads it.
286 
287 ## Enable array selection in Fides reader
288 
289 You can now choose which arrays should be loaded with the **Fides reader**. The arrays available to load must still be specified in the JSON data model file, but you can now enable/disable loading those arrays through the GUI.

397 ## Handle semicolon and hash characters in the Recent Files menu
398 
399 ParaView would crash on startup if a file was opened with a semicolon ";" in the filename. The `Recent Files` menu and saved settings uses ";" as a record separator, and hash "#" as a server name indicator. These characters are now encoded in the settings file to avoid incorrect interpretation.

401 # Catalyst
402 
403 Updates listed below refer to ParaView's Catalyst 2 implementation.
404 
405 ## Catalyst pipeline selection
406 
407 Support for which Catalyst pipelines to execute has been added. This includes Python script pipelines as well as precompiled pipelines. The Python script pipelines are configured with:
408 
409 ```
410 node['catalyst/scripts/aname'] = afile.py
411 ```
412 
413 where `aname` is the name of the pipeline. Precompiled pipelines are configure with:
414 
415 ```
416 node["catalyst/pipelines/0/type"] = "io"
417 node["catalyst/pipelines/0/filename"] = "foo-%04ts.vtpd"
418 node["catalyst/pipelines/0/channel"] = "input"
419 ```
420 
421 where `0` is the name of the pipeline.
422 
423 Pipelines can be selected by listing them under the `catalyst/state/pipelines` node such as
424 
425 ```
426 node['catalyst/state/pipelines/0'] = 'aname'
427 ```
428 
429 Note that the value of the node (`aname`) is what matters and not the name of the node (`0`). It is possible to turn off all of the pipelines by making pipelines a leaf node such as:
430 
431 ```
432 node['catalyst/state/pipelines'] = 0
433 ```
434 
435 If `catalyst/state/pipelines does not exist`, Catalyst will, by default, execute all pipelines.
436 
437 ## Custom `catalyst_results` in Catalyst scripts
438 
439 You can now define custom `catalyst_results(info)` functions in Catalyst scripts which will be executed when the simulation calls `catalyst_results`. The Conduit node passed as input to the call from the simulation side is available via `info.catalyst_params`.
440 
441 ## Catalyst direct access to Conduit during execution
442 
443 The input parameters of `catalyst_execute` are now accessible as a read-only Conduit node inside a catalyst script. You can now perform any kind of analysis directly on the data passed by the simulation.
444 
445 ## Extractors for Catalyst steering
446 
447 Catalyst's "steering" capabilities can send data back to the simulation in order to "steer" it. In order to use the steering, the simulation used to send an XML file specifying a `vtkSteeringDataGenerator` proxy and properties to be filled by Paraview Catalyst.
448 
449 While this still works, we introduce Catalyst **steering extractors**, a new kind of extractor acting as "sinks" in the visualization pipeline. Calling `catalyst_results` from the simulation will trigger these sinks, and their input dataset will be serialized to Conduit's mesh blueprint and sent back to the simulation in the results Conduit node. This special kind of extractor needs to be created in the Catalyst Python pipeline script. They are **not** associated to a trigger and a writer like other extractors are. They are created by the extracts controller but managed externally by the in situ helper.
450 
451 Example Catalyst pipeline script:
452 
453 ```python
454 # Data is provided by the simulation.
455 producer = TrivialProducer(registrationName="grid")
456 
457 # Perform a reduction on the input data.
458 sliced = Slice(Input=producer, SliceType="Plane")
459 
460 # Create a steering extractor from the slice.
461 steering = CreateExtractor('steering', sliced, registrationName='steered_slice')
462 ```
463 
464 This script is executed as usual when `catalyst_execute` is called, but the extraction happens only when the simulation calls `catalyst_results`. This feature can be used to perform a reduction on input data, for example, and get the result in the simulation directly.

467 # Developer notes
468 
469 ## Fix the behavior of vtkCLIOptions::SetStopOnUnrecognizedArgument
470 
471 `SetStopOnUnrecognizedArgument` was behaving in an inverted way according to its name and documentation. This was fixed and documentation was clarified.
472 
473 Now, `vtkCLIOptions::SetStopOnUnrecognizedArgument(true)` will indeed stop the parsing on unrecognized arguments.
474 
475 See https://github.com/CLIUtils/CLI11/issues/1052 for more info.
476 
477 ## Building against external VTK
478 
479 Support has been added for compiling against and using an external VTK. The version (git hash) of the external VTK has to match the version of the VTK sub-module included in the version of ParaView being compiled. While this is not enforced, it is likely that you would get a compilation error or a crash if you try to use ParaView with a differnet version of VTK than the one it was tested with (the version of the VTK submodule).
480 
481 This feature is enabled with the `PARAVIEW_USE_EXTERNAL_VTK` CMake option enabled. `VTK_DIR` can be used to specify the location for the config file of the VTK install, such as: `<vtk_install_dir>/lib/cmake/vtk-9.3`.
482 
483 Depending on what is enabled in ParaView, certain features need to be provided by the external VTK. The features required by ParaView but not provided by the VTK install, would be printed by CMake at configure time in a format that can be copied and pasted in a VTK reconfigure command that would compile those features in VTK.
484 
485 ## Specification of backend when using VTK-m w/ Kokkos
486 
487 This change propogates build flags from ParaView to VTK and affects how accelerated filters are built using VisKores (previously `VTK-m`). It does away with the requirement of using flags like `PARAVIEW_USE_HIP` when `VTK-m` is built with the necessary Kokkos backend. This also future-proofs the specification for backends. Two new flags are introduced `PARAVIEW_USE_KOKKOS` and `PARAVIEW_KOKKOS_BACKEND` -- the first flag is considered `ON` automatically if the backend flag is provided.
488 
489 Here is how you can build with different backends
490 
491 * `PARAVIEW_USE_CUDA` -- use the native CUDA support from VTK-m
492 * `PARAVIEW_USE_KOKKOS + PARAVIEW_KOKKOS_BACKEND=CUDA` or `PARAVIEW_KOKKOS_BACKEND=CUDA` will use Kokkos with CUDA backend
493 * `PARAVIEW_USE_KOKKOS + PARAVIEW_KOKKOS_BACKEND=SYCL` or `PARAVIEW_KOKKOS_BACKEND=SYCL` will use Kokkos with SYCL backend
494 * `PARAVIEW_USE_KOKKOS + PARAVIEW_KOKKOS_BACKEND=HIP` or `PARAVIEW_KOKKOS_BACKEND=HIP` will use Kokkos with HIP backend
495 * `PARAVIEW_USE_KOKKOS` will use Kokkos with Serial backend

575 ## `vtkPVVersion`
576 
577 Most of the version macros have been relocated to `vtkPVVersionQuick.h`. The new `PARAVIEW_VERSION_EPOCH` macro is defined as the actual `PARAVIEW_VERSION_PATCH` value for release-track development and a "high" value for future ParaView release development (i.e., the next minor version bump). The actual patch number, and `PARAVIEW_VERSION_FULL` values are available in `vtkPVVersion.h`. A new `PARAVIEW_VERSION_NUMBER_QUICK` macro provides the ParaView version as a comparable value with `PARAVIEW_VERSION_CHECK` using the `PARAVIEW_VERSION_EPOCH` instead of `PARAVIEW_VERSION_PATCH`. The intent is to reduce the rebuild frequency needed for the nightly build version bump for the in-development source.

601 ## Introducing Emulated Time Algorithm using LiveSources
602 
603 LiveSource capabilities have been expanded with a new emulated time algorithm.
604 
605 Once you've inherited your temporal algorithm from `vtkEmulatedTimeAlgorithm`, you can use the LiveSource concept to synchronize the algorithm's timesteps on a "Real Time" interval (in seconds). It will skip frames if the pipeline processing does not keep up.
606 
607 An example scenario for this algorithm would be to replay a LiveSource stream from a recorded file while maintaining the original conditions and playback speed.
608 
609 To implement this behavior, inherit your algorithm from `vtkEmulatedTimeAlgorithm` and define the proxy as a LiveSource with the `emulated_time` attribute.
610 ```xml
611  <Hints>
612  <LiveSource interval="100" emulated_time="1" />
613  </Hints>
614 ```
