1 # Plugin Howto {#PluginHowto}
2 
3 ## Introduction
4 
5 ParaView comes with plethora of functionality bundled in: several readers,
6 multitude of filters, different types of views, etc. However, it is not
7 uncommon for developers to want to add new functionality to ParaView to, for
8 example, add support to their new file format or incorporate a new filter into
9 ParaView. ParaView makes it possible to add new functionality by using an
10 extensive plugin mechanism.
11 
12 Plugins can be used to extend ParaView in several ways:
13 
14  * Add new readers, writers, filters
15  * Add custom GUI components such as toolbar buttons to perform common tasks
16  * Add new views in for display data
17 
18 Examples for different types of plugins are provided with the ParaView source
19 under `Examples/Plugins/`.
20 
21 This document has major sections:
22 
23  * First section covers how to use existing plugins in ParaView.
24  * Second section contains information for developers about writing new
25  plugins for ParaView.
26 
27 ## Using Plugins
28 
29 ### Types of plugins
30 
31 Plugins are distributed as shared libraries (`*.so` on Unix and macOS, and
32 `*.dll` on Windows). For a plugin to be loadable in ParaView, it must be built
33 with the same version of ParaView as it is expected to be deployed on. Plugins
34 can be classified into two broad categories:
35 
36  * *Server-side plugins*: These are plugins that extend the algorithmic
37  capabilities for ParaView. For example, new filters, readers or writers.
38  Since ParaView processes data on the server-side, these plugins need to be
39  loaded on the server.
40  * *Client-side plugins*: These are plugins that extend the ParaView GUI
41  including property panels for new filters, toolbars or views. These plugins
42  need to be loaded on the client.
43 
44 Oftentimes a plugin has both server-side as well as client-side components to
45 it. For example, a plugin that adds a new filter and a property panel that goes
46 with that filter. Such plugins need to be loaded both on the server as well as
47 the client.
48 
49 Generally, users don't have to worry whether a plugin is a server-side or
50 client-side plugin. Simply load the plugin on the server as well as the client.
51 ParaView will include relevant components from plugin on each of the processes.
52 
53 ### Loading plugins
54 
55 There are four ways for loading plugins:
56 
57  * *Using the GUI* (`Plugin Manager`)
58  - Plugins can be loaded into ParaView using the `Plugin Manager` accessible
59  from the `Tools | Manage Plugins/Extensions` menu. The `Plugin Manager`
60  has two sections for loading client plugins and server plugins (shown
61  only when connected to a server). To load a plugin on the client and
62  server side, simply browse to the plugin shared library. If the plugin is
63  loaded successfully, it will appear in the list of loaded plugins. The
64  `Plugin Manager` also lists the paths it searched to load plugins
65  automatically.
66  - The `Plugin Manager` remembers all loaded plugins across ParaView
67  instances, so once a plugin is loaded once, it will appear in the future
68  (unloaded).
69  - You can set up ParaView to automatically load the plugin at startup (for
70  client plugins) or on connecting to the server (for server plugins) by
71  checking the "Auto Load" checkbox on a loaded plugin.
72 
73 ![Plugin Manager when not connected to a remote server, showing loaded plugins on the local site.](images/LocalPlugin_Manager.png)
74 ![Plugin Manager when connected to a server showing loaded plugins on the local as well as remote sites.](images/RemotePlugin_Manager.png)
75 
76  * *Using environment variable* (Auto-loading plugins)
77  - In order to have ParaView automatically load a set of plugins on startup,
78  one can use the `PV_PLUGIN_PATH` environment variable. `PV_PLUGIN_PATH`
79  may be used to list a set of directories (separated by colon (`:`) for
80  Unix platforms or semi-colon (`;`) on Windows) which ParaView will search
81  on startup to load plugins. This environment variable needs to be set on
82  both the client and server sides to load their respective plugins. Note
83  that plugins in PV_PLUGIN_PATH are always auto-loaded irrespective of the
84  status of the `Auto Load` checkbox in the `Plugin Manager`.
85  - Finer control can be used using the `PV_PLUGIN_CONFIG_FILE` environment
86  variable. `PV_PLUGIN_CONFIG_FILE` can be used to list a set of XML plugin
87  configuration files (separated by colon (`:`) on Unix platforms or
88  semi-colon (`;`) on Windows). ParaView will read these files on startup
89  to load specified plugins. The XML plugin configuration file format looks
90  like this:
91 
92 ```xml
93 <?xml version="1.0"?>
94 <Plugins>
95  <Plugin name="MyPlugin" filename="/absolute/path/to/libMyPlugin.so"/>
96 </Plugins>
97 ```
98 
99  Plugins listed this way will always be loaded, irrespective of the status
100  of the `Auto Load` checkbox in the `Plugin Manager`.
101  * *Using the plugin file `.plugins`* (Make plugins available and possibly
102  Auto-load plugins)
103  - Plugins that are listed in the `.plugins` file on the client and server
104  will automatically be listed in the `Plugin Manager`, and may optionally
105  be auto loaded. ParaView creates its own `.plugins` file listing plugins
106  known during its build and uses it as the default. An example `.plugins`
107  file, auto loading H5PartReader, looks like this:
108 
109 ```xml
110 <?xml version="1.0"?>
111 <Plugins>
112  <Plugin name="Moments" auto_load="0"/>
113  <Plugin name="PrismPlugin" auto_load="0"/>
114  <Plugin name="PointSprite_Plugin" auto_load="0"/>
115  <Plugin name="pvblot" auto_load="0"/>
116  <Plugin name="SierraPlotTools" auto_load="0"/>
117  <Plugin name="H5PartReader" auto_load="1"/>
118 </Plugins>
119 ```
120 
121  * *Default search paths*
122  - Recognized locations are:
123  * A `plugins` subdirectory under the `paraview-X.Y` directory in the
124  library path (usually `lib` on Unix platforms and `bin` on Windows).
125  * A `plugins` subdirectory in the user's home directory. On Unix
126  platforms, `$HOME/.config/ParaView/ParaView<version>/Plugins`. On
127  Windows `%APPDATA$\ParaView\ParaView<version>\Plugins`. (XXX: no
128  evidence of an implementation backing this search path).
129

153 # Writing plugins
154 
155 This section covers writing and compiling different types of Plugins. To create
156 a plugin, one must have their own build of ParaView. Binaries downloaded from
157 www.paraview.org do not include necessary header files or import libraries
158 (where applicable) for compiling plugins.
159 
160 The `CMakeLists.txt` file used in all following examples start off with the
161 following code:
162 
163 ```cmake
164 # ParaView requires CMake 3.8 in order to be used.
165 cmake_minimum_required(VERSION 3.8)
166 project(myplugin)
167 
168 find_package(ParaView REQUIRED)
169 ```
170 
171 Where CMake will ask for the `ParaView_DIR` which you point to the ParaView
172 build or install tree you would to build your with.
173 
174 ## Exposing an Existing Filter
175 
176 Sometimes, the filter that one wants to add to ParaView is already available in
177 VTK, it's just not exposed through the ParaView GUI. This is the easiest type
178 of plugin to create. There are two options:
179 
180  1. setup the plugin using only an XML file; and
181  2. actually compile the plugin into a shared library.
182 
183 The first option is the easiest, but the second option will prepare you for
184 creating a custom filter in the future as the process is nearly identical.
185 
186 ## Adding a New Filter
187 
188 It is also possible to add new filters to ParaView. The filter has to be a
189 VTK-based algorithm, written as following the standard procedures for writing
190 VTK algorithms. Generally for such cases where we are adding a new VTK class to
191 ParaView (be it a filter, reader or a writer), we need to do the following
192 tasks:
193 
194  * Write a *Server Manager Configuration XML* which describes the `Proxy`
195  interface for the filter. Basically, this defines the interface for the
196  client to create and modify instances of the new class on the server side.
197  Please refer to the [ParaView Guide][] for details about writing these
198  server-manager XMLs.
199  * Write a configuration XML for the GUI to make ParaView GUI aware of this
200  new class, if applicable. For filters, this is optional, since ParaView
201  automatically recognizes filters added through plugins and lists them in
202  the *Alphabetical* sub-menu. One may use the GUI configuration XML to add
203  the new filter to a specific category in the *Filters* menu, or add a new
204  category. For readers and writers, this is required since ParaView GUI
205  needs to know what extensions your reader/writer supports etc.
206 
207 ## Examples
208 
209 ### XML Plugins
210 
211 If you have not built ParaView from source, using an XML plugin is your only
212 option.
213 
214 First, a server manager XML for the filter is required. The GUI XML to add the
215 filter to any specific category is optional.
216 
217 For example, let's say we simply want to expose the `vtkCellDerivatives` filter
218 in VTK. Then first, we'll write the server manager configuration XML (call it
219 `CellDerivatives.xml`), similar to what we would have done for adding a new
220 filter.
221 
222 ```xml
223 <ServerManagerConfiguration>
224  <ProxyGroup name="filters">
225  <SourceProxy name="MyCellDerivatives" class="vtkCellDerivatives" label="My Cell Derivatives">
226  <Documentation
227  long_help="Create point attribute array by projecting points onto an elevation vector."
228  short_help="Create a point array representing elevation.">
229  </Documentation>
230  <InputProperty
231  name="Input"
232  command="SetInputConnection">
233  <ProxyGroupDomain name="groups">
234  <Group name="sources"/>
235  <Group name="filters"/>
236  </ProxyGroupDomain>
237  <DataTypeDomain name="input_type">
238  <DataType value="vtkDataSet"/>
239  </DataTypeDomain>
240  </InputProperty>
241  </SourceProxy>
242  </ProxyGroup>
243 </ServerManagerConfiguration>
244 ```
245 
246 At this point, we can stop and use the plugin in ParaView by loading the XML
247 file directly into the plugin manager.
248 
249 Please note that if you are writing the XML for a filter that takes just one
250 input, you *must* set the `name` attribute for the `InputProperty` XML element
251 to `Input`. If you do not, then the filter will not be displayed properly in
252 ParaView's pipeline browser.
253 
254 ### Compiling into a Shared Library
255 
256 If you have built ParaView from source, it is possible to compile the plugin
257 into into a shared library. To do this, we can use the following top-level:
258 `CMakeLists.txt`:
259 
260 ```cmake
261 # Standard CMake boilerplate. ParaView's `find_package` requires at least 3.8.
262 cmake_minimum_required(VERSION 3.8)
263 project(sharedlibrary)
264 
265 # These two lines are required in order to set up installation directories
266 # (which also control build directory locations) and enable shared builds
267 # (CMake's default is for a static build).
268 include(GNUInstallDirs)
269 set(BUILD_SHARED_LIBS ON)
270 
271 # Find ParaView. This will bring in ParaView's CMake API and imported targets.
272 find_package(ParaView REQUIRED)
273 
274 # Scan the plugin file in order to set up internal data structures for building
275 # plugins.
276 paraview_plugin_scan(
277  # The `paraview.plugin` file describing the plugin.
278  PLUGIN_FILES "${CMAKE_CURRENT_SOURCE_DIR}/Plugin/paraview.plugin"
279  # A result variable for the (enabled) plugins found during the scan.
280  PROVIDES_PLUGINS plugins
281  # Enable plugins during this scan by default.
282  ENABLE_BY_DEFAULT ON)
283 
284 # Build the plugins discovered during the scan.
285 paraview_plugin_build(
286  PLUGINS ${plugins})
287 ```
288 
289 The mentioned `paraview.plugin` file describes the plugin to the build system:
290 
291 ```cmake
292 NAME
293  CellDerivatives
294 DESCRIPTION
295  Expose the vtkCellDerivatives class to ParaView.
296 REQUIRES_MODULES
297  # This module provides the `vtkCellDerivatives` filter.
298  VTK::FiltersGeneral
299 ```
300 
301 In the `Plugin` directory (beside the `paraview.plugin` file), the plugin is
302 given the information it needs to build:
303 
304 ```cmake
305 paraview_add_plugin(CellDerivatives
306  VERSION "1.0"
307  SERVER_MANAGER_XML CellDerivatives.xml)
308 ```
309 
310 Then using CMake, one can build a plugin for this new filter. We can now load
311 the plugin through the plugin manager by selecting the created `.so` or `.dll`
312 file.
313 
314 ### Qt resource plugins
315 
316 Similarly compiled Qt resources (`*.bqrc`) can be loaded at runtime. A `.bqrc`
317 file is a binary file containing resources which can include icons, the GUI
318 configuration XML for adding categories, etc. A `.bqrc` can be made from a
319 `.qrc` by running the `rcc` utility provided by Qt:
320 
321 ```sh
322 rcc -binary -o myfile.bqrc myfile.qrc
323 ```

325 ### Adding a New Filter
326 
327 For this example, refer to `Examples/Plugins/ElevationFilter` in the ParaView
328 source. Let's say we have written a new `vtkMyElevationFilter`
329 (`vtkMyElevationFilter.{h,cxx}`), which extends the functionality of the
330 `vtkElevationFilter` and we want to package that as a plugin for ParaView. For
331 starters, we simply want to use this filter in ParaView (e.g., not doing
332 anything fancy with *Filters* menu categories). As described, we need to write
333 the server manager configuration XML (`MyElevationFilter.xml`). Once that's
334 done, we write a `CMakeLists.txt` file to package this into a plugin.
335 
336 This `CMakeLists.txt` needs to include the following lines:
337 
338 ```cmake
339 cmake_minimum_required(VERSION 3.8)
340 project(newfilter)
341 
342 include(GNUInstallDirs)
343 set(BUILD_SHARED_LIBS ON)
344 
345 find_package(ParaView REQUIRED)
346 
347 paraview_plugin_scan(
348  PLUGIN_FILES "${CMAKE_CURRENT_SOURCE_DIR}/Plugin/paraview.plugin"
349  PROVIDES_PLUGINS plugins
350  ENABLE_BY_DEFAULT ON)
351 
352 paraview_plugin_build(
353  PLUGINS ${plugins})
354 ```
355 
356 The referenced `paraview.plugin` file contains:
357 
358 ```cmake
359 NAME
360  ElevationFilter
361 DESCRIPTION
362  An example paraview plugin containing server manager XML and the server
363  manager classes to build. This plugin can be loaded on the server side.
364 REQUIRES_MODULES
365  VTK::CommonCore
366  VTK::FiltersCore
367 ```
368 
369 And the `CMakeLists.txt` file beside it contains:
370 
371 ```cmake
372 paraview_add_plugin(ElevationFilter
373  VERSION "1.0"
374  MODULES ElevationFilters
375  MODULE_FILES "${CMAKE_CURRENT_SOURCE_DIR}/ElevationFilters/vtk.module")
376 ```
377 
378 Because we are building our own custom filter, it needs to be a VTK module in
379 order to support having its information available to the XML code. First, the
380 module is declared in a `vtk.module` file:
381 
382 ```cmake
383 NAME
384  ElevationFilters
385 DEPENDS
386  VTK::FiltersCore
387 PRIVATE_DEPENDS
388  VTK::CommonCore
389 ```
390 
391 And then the module is built with its associated server manager XML file
392 attached to the module.
393 
394 ```cmake
395 set(classes
396  vtkMyElevationFilter)
397 
398 vtk_module_add_module(ElevationFilters
399  CLASSES ${classes})
400 
401 paraview_add_server_manager_xmls(
402  XMLS MyElevationFilter.xml)
403 ```
404 
405 Then using CMake, one can build a plugin for this new filter. Once this plugin
406 is loaded the filter will appear under the *Alphabetical* list in the *Filters*
407 menu. Note that there will be two libraries in the resulting directory. Be sure
408 to load the `ElevationFilter` one which is the plugin, not the
409 `ElevationFilters` module library.
410 
411 ### Filters with Multiple Input Ports
412 
413 If a filter requires multiple input ports, there are two options:
414 
415  1. Create helper functions in the VTK filter such as `SetYourInputName` which
416  deal with addressing the VTK pipeline in the C++ code; and
417  2. Address/access the input connection by number in the XML. The `port_index`
418  property specifies which input connection the particular input will be
419  connected to. The `SetInputConnection` function is the command that will
420  actually be called with this `port_index` to setup the pipeline.
421 
422 An example XML file for a filter with multiple inputs is below. The filter
423 takes three `vtkPolyData` objects as input.
424 
425 ```xml
426 <ServerManagerConfiguration>
427  <ProxyGroup name="filters">
428  <SourceProxy name="LandmarkTransformFilter" class="vtkLandmarkTransformFilter" label="LandmarkTransformFilter">
429  <Documentation
430  long_help="Align two point sets using vtkLandmarkTransform to compute the best transformation between the two point sets."
431  short_help="vtkLandmarkTransformFilter.">
432  </Documentation>
433 
434  <InputProperty
435  name="SourceLandmarks"
436  port_index="0"
437  command="SetInputConnection">
438  <ProxyGroupDomain name="groups">
439  <Group name="sources"/>
440  <Group name="filters"/>
441  </ProxyGroupDomain>
442  <DataTypeDomain name="input_type">
443  <DataType value="vtkPolyData"/>
444  </DataTypeDomain>
445  <Documentation>
446  Set the source data set. This data set that will move towards the target data set.
447  </Documentation>
448  </InputProperty>
449 
450  <InputProperty
451  name="TargetLandmarks"
452  port_index="1"
453  command="SetInputConnection">
454  <ProxyGroupDomain name="groups">
455  <Group name="sources"/>
456  <Group name="filters"/>
457  </ProxyGroupDomain>
458  <DataTypeDomain name="input_type">
459  <DataType value="vtkPolyData"/>
460  </DataTypeDomain>
461  <Documentation>
462  Set the target data set. This data set will stay stationary.
463  </Documentation>
464  </InputProperty>
465 
466  <InputProperty
467  name="SourceDataSet"
468  port_index="2"
469  command="SetInputConnection">
470  <ProxyGroupDomain name="groups">
471  <Group name="sources"/>
472  <Group name="filters"/>
473  </ProxyGroupDomain>
474  <DataTypeDomain name="input_type">
475  <DataType value="vtkPolyData"/>
476  </DataTypeDomain>
477  <Documentation>
478  Set the source data set landmark points.
479  </Documentation>
480  </InputProperty>
481 
482  <Hints>
483  <!-- see below for what options to put here -->
484  </Hints>
485 
486  </SourceProxy>
487  </ProxyGroup>
488 </ServerManagerConfiguration>
489 ```
490 
491 To set the inputs in ParaView, simply select one of the inputs in the *Pipeline
492 Browser* and then select the filter from the *Filters* menu. This will open a
493 dialog box which will allow you to specify which object to connect to each
494 input port.

496 ### Adding *Categories* to the *Filters* Menu
497 
498 Now suppose we want to add a new category to the *Filters* menu, called
499 *Extensions* and then show this filter in that menu. In that case we need to
500 add a hint to the XML file that tells ParaView what category to display this
501 filter in. In this case, the `Hints` element of the XML file can contain:
502 
503 ```xml
504 <Hints>
505  <ShowInMenu category="Extensions" />
506 </Hints>
507 ```
508 
509 If the name of the category is same as an already existing category such as
510 *Data Analysis*, then the filter gets added to the existing category.
511 
512 ### Adding Icons
513 
514 You can see that some filters in the *Filters* menu (e.g., *Clip*) have icons
515 associated with them. It's possible for the plugin to add icons for filters it
516 adds as well. For that you need to write a Qt resource file (say
517 `MyElevation.qrc`) as follows:
518 
519 ```xml
520 <RCC>
521  <qresource prefix="/MyIcons" >
522  <file>MyElevationIcon.png</file>
523  </qresource>
524 </RCC>
525 ```
526 
527 To use the icon for a filter in the pipeline add the following hint to the
528 server manager XML.
529 
530 ```xml
531 <Hints>
532  <ShowInMenu icon=":/MyIcons/MyElevationIcon.png" />
533 </Hints>
534 ```
535 
536 Finally, the plugin's `CMakeLists.txt` file much change to include our
537 `MyElevation.qrc` file as follows:
538 
539 ```cmake
540 paraview_add_plugin(ElevationFilter
541  VERSION "1.0"
542  MODULES ElevationFilters
543  MODULE_FILES "${CMAKE_CURRENT_SOURCE_DIR}/ElevationFilters/vtk.module"
544  UI_RESOURCES MyElevation.qrc)
545 ```

547 ### Adding GUI Parameters
548 
549 Simply add these in the server manager XML to expose parameters of the filter
550 to the ParaView user.
551 
552 #### Integer Property
553 
554 This property appears as a text box.
555 
556 ```xml
557 <IntVectorProperty
558  name="bStartByMatchingCentroids"
559  command="SetbStartByMatchingCentroids"
560  number_of_elements="1"
561  default_values="1">
562 </IntVectorProperty>
563 ```
564 
565 #### Boolean Property
566 
567 This property appears as a check box control. A boolean property uses the
568 `IntVectorProperty` with an extra line (`BooleanDomain`) indicating this should
569 be a check box rather than a text field.
570 
571 ```xml
572 <IntVectorProperty
573  name="bStartByMatchingCentroids"
574  command="SetbStartByMatchingCentroids"
575  number_of_elements="1"
576  default_values="1">
577  <BooleanDomain name="bool"/>
578 </IntVectorProperty>
579 ```
580 
581 #### String Property
582 
583 This property appears as a text box.
584 
585 ```xml
586 <StringVectorProperty
587  name="YourStringVariable"
588  command="SetYourStringVariable"
589  number_of_elements="1"
590  default_values="1">
591 </StringVectorProperty>
592 ```
593 
594 #### Double Property
595 
596 This property appears as a text box.
597 
598 ```xml
599 <DoubleVectorProperty
600  name="YourDoubleVariable"
601  command="SetYourDoubleVariable"
602  number_of_elements="1"
603  default_values="1">
604 </DoubleVectorProperty>
605 ```
606 
607 #### Multi-Value Double Property
608 
609 This property appears as a text box.
610 
611 ```xml
612 <DoubleVectorProperty
613  name="YourDoubleVectorVariable"
614  command="SetYourDoubleVectorVariable"
615  number_of_elements="3"
616  default_values="1.0 0.0 0.0">
617 </DoubleVectorProperty>
618 ```
619 
620 #### Double Property Slider
621 
622 This creates a slider that ranges from `0.0` to `1.0`.
623 
624 ```xml
625 <DoubleVectorProperty
626  name="PercentToRemove"
627  command="SetPercentToRemove"
628  number_of_elements="1"
629  default_values="0.1">
630  <DoubleRangeDomain name="range" min="0.0" max="1.0" />
631 </DoubleVectorProperty>
632 ```
633 
634 #### Drop Down List
635 
636 This creates a drop down list with 3 choices. The values associated with the
637 choices may be specified.
638 
639 ```xml
640 <IntVectorProperty
641  name="TransformMode"
642  command="SetTransformMode"
643  number_of_elements="1"
644  default_values="1">
645  <EnumerationDomain name="enum">
646  <Entry value="6" text="RigidBody"/>
647  <Entry value="7" text="Similarity"/>
648  <Entry value="12" text="Affine"/>
649  </EnumerationDomain>
650  <Documentation>
651  This property indicates which transform mode will be used.
652  </Documentation>
653 </IntVectorProperty>
654 ```
655

657 
658 This creates a list that lets you choose among the input arrays of the input of

660 
661 ```xml
662 <StringVectorProperty

671  attribute_type="Scalars"
672  input_domain_name="inputs_array">
673  <RequiredProperties>
674  <Property name="Input" function="Input" />
675  </RequiredProperties>

685

750 #### Adding a Reader
751 
752 Adding a new reader through a plugin is similar to adding a filter. The only
753 difference is that we do not need to specify what category the reader should be
754 added to in the GUI. For the latest version of ParaView we do not need to
755 specify anything special for the GUI as all of the details of the reader are
756 available in the XML proxy definition of the reader. For ParaView version 4.0.1
757 and earlier we need the XML to define what file extensions this reader can
758 handle. This XML (`MyReaderGUI.xml`) looks like this:
759 
760 ```xml
761 <ParaViewReaders>
762  <Reader name="MyPNGReader" extensions="png"
763  file_description="My PNG Files">
764  </Reader>
765 </ParaViewReaders>
766 ```
767 
768 An example `MyPNGReader.xml` is shown below. In almost all cases you must have
769 a `SetFileName` function property. You are free to have other properties as
770 well, as with a standard (non-reader) filter. Also, the `Hints` section is needed
771 in order to associate the file extension with the reader on the client. The
772 `ReaderFactory` hint is what the client uses to identify readers from sources.
773 
774 ```xml
775 <ServerManagerConfiguration>
776  <ProxyGroup name="sources">
777  <SourceProxy name="MyPNGReader" class="vtkMyPNGReader" label="PNGReader">
778  <Documentation
779  long_help="Read a PNG file."
780  short_help="Read a PNG file.">
781  </Documentation>
782  <StringVectorProperty
783  name="FileName"
784  animateable="0"
785  command="SetFileName"
786  number_of_elements="1">
787  <FileListDomain name="files"/>
788  <Documentation>
789  This property specifies the file name for the PNG reader.
790  </Documentation>
791  </StringVectorProperty>
792  <Hints>
793  <ReaderFactory extensions="png"
794  file_description="PNG File Format" />
795  </Hints>
796  </SourceProxy>
797  </ProxyGroup>
798 </ServerManagerConfiguration>
799 ```
800 
801 The CMake code for a reader plugin uses the same structure as the filter
802 example. The only likely difference is that the plugin should also pass
803 `REQUIRED_ON_SERVER` to `paraview_add_plugin` since the server side needs the
804 reader available for its use.
805 
806 If you want your reader to work correctly with a file series, please refer to [[Animating legacy VTK file series#Making custom readers work with file series|file series animation]] for details.
807 
808 Once you generate the project using CMake and compile the project, in ParaView
809 go to *Tools > Manage Plugins/Extensions*. Under *Local Plugins*, click *Load
810 New* and browse for the shared library file you just created. You should now
811 see your new file type in the *Files of type* list in the *Open file* dialog.

826 #### Adding Customizations for Properties Panel
827 
828 <!-- TODO link to external page -->
829 
830 [[ParaView/Properties Panel|Properties Panel]] is the primary panel in ParaView
831 used to change the parameters for visualization modules and displays. Plugins
832 can provide new types of [`pqPropertyWidget`][pqPropertyWidget] subclasses that
833 can be used to control properties/property groups on this Properties panel.
834 
835 To register a new `pqPropertyWidget` subclass to be associated with a
836 particular widget type for a property (`vtkSMProperty`), use the following
837 CMake code in your plugin:
838 
839 ```cmake
840 paraview_plugin_add_property_widget
841  KIND kind
842  TYPE my_property_widget_type
843  CLASS_NAME ClassName
844  INTERFACES interfaces
845  SOURCES sources)
846 
847 paraview_add_plugin(propwidget
848  VERSION "1.0"
849  UI_INTERFACES ${interfaces}
850  SOURCES ${sources})
851 ```
852 
853 The `KIND` argument must be one of `WIDGET`, `GROUP_WIDGET`, or
854 `WIDGET_DECORATOR`. For a `vtkSMProperty`, `WIDGET` is required.
855 
856 The `CLASS_NAME` argument must refer to a `pqPropertyWidget` subclass with a
857 constructor with the following signature:
858 
859 ```cpp
860 ClassName(vtkSMProxy *smproxy, vtkSMProperty *smproperty, QWidget *parentObject)
861 ```
862 
863 The `TYPE` argument specifies the string that will be used in the server
864 manager XML as the value for the `panel_widget` attribute to request creation
865 of this widget for a `vtkSMProperty` subclass.
866 
867 To register a new `pqPropertyWidget` subclass to be associated with a
868 particular widget type for a property group (`vtkSMPropertyGroup`), use
869 `GROUP_WIDGET` for the `KIND` argument. The referenced `CLASS_NAME` must
870 subclass `pqPropertyWidget` and have a constructor with the signature:
871 
872 ```cpp
873 ClassName(vtkSMProxy *smproxy, vtkSMPropertyGroup *smgroup, QWidget *parentObject);
874 ```
875 
876 As before, the `TYPE` specifies the string that will be used in the server
877 manager XML as the value for the `panel_widget` attribute on a `PropertyGroup`
878 element to request creation of this widget for that group.
879 
880 Another mechanism for adding customizations for *Properties* panel is to
881 provide [`pqPropertyWidgetDecorator`][pqPropertyWidgetDecorator] subclasses to
882 add custom control logic for widgets on the panel.
883 
884 Decorators use the `WIDGET_DECORATOR` argument to `KIND`.
885 
886 The `CLASS_NAME` must point to a `pqPropertyWidgetDecorator` subclass and the
887 `TYPE` is the string name used to request the creation of the decorator in the
888 server manager XML as described [[ParaView/Properties Panel|here]].
889 
890 An example for customizing the Properties panel can be found in the ParaView
891 source under `Examples/Plugins/PropertyWidgets`.

921 #### Adding a Toolbar
922 
923 Filters, reader, and writers are by far the most common ways for extending
924 ParaView. However, ParaView plugin functionality goes far beyond that. The
925 following sections cover some of these advanced plugins that can be written.
926 
927 Applications use toolbars to provide easy access to commonly used
928 functionality. It is possible to have plugins that add new toolbars to
929 ParaView. The plugin developer implements his own C++ code to handle the
930 callback for each button on the toolbar. Hence one can do virtually any
931 operation using the toolbar plugin with some understanding of the ParaView
932 Server Manager framework and the ParaView GUI components.
933 
934 Please refer to `Examples/Plugins/SourceToolbar` for this section. There we are
935 adding a toolbar with two buttons to create a sphere and a cylinder source. For
936 adding a toolbar, one needs to implement a subclass for
937 [`QActionGroup`][QActionGroup] which adds the [`QAction`][QAction]s for each of
938 the toolbar button and then implements the handler for the callback when the
939 user clicks any of the buttons. In the example `SourceToobarActions.{h,cxx}` is
940 the `QActionGroup` subclass that adds the two tool buttons.
941 
942 To build the plugin, the `CMakeLists.txt` file is:
943 
944 ```cmake
945 # This is a macro for adding QActionGroup subclasses automatically as toolbars.
946 paraview_plugin_add_action_group(
947  CLASS_NAME SourceToolbarActions
948  GROUP_NAME "ToolBar/SourceToolbar"
949  INTERFACES interfaces
950  SOURCES sources)
951 
952 # Now create a plugin for the toolbar. Here we pass the `interfaces` and
953 # `sources` returned by the above call.
954 paraview_add_plugin(SourceToolbar
955  VERSION "1.0"
956  UI_INTERFACES ${interfaces}
957  SOURCES ${sources}
958  SourceToolbarActions.cxx)
959 ```
960 
961 For the `GROUP_NAME`, we are using `ToolBar/SourceToolbar`; here `ToolBar` is a
962 keyword which implies that the action group is a toolbar (and shows up under
963 *View > Toolbars* menu) with the name `SourceToolbar`. When the plugin is
964 loaded, this toolbar will show up with two buttons.
965 
966 #### Adding a Menu
967 
968 Adding a menu to the menu bar of the main window is almost identical to adding
969 a toolbar. The only difference is that you use the keyword `MenuBar` in lieu
970 of `ToolBar` in the `GROUP_NAME` of the action group. So if you change the
971 `paraview_plugin_add_action_group` command above to the following, the plugin
972 will add a menu titled `MyActions` to the menu bar.
973 
974 ```cmake
975 paraview_plugin_add_action_group(
976  CLASS_NAME SourceToolbarActions
977  GROUP_NAME "MenuBar/MyActions"
978  INTERFACES interfaces
979  SOURCES sources)
980 ```
981 
982 If you give the name of an existing menu, then the commands will be added to
983 that menu rather than create a new one. So, for example, if the `GROUP_NAME`
984 is `MenuBar/File`, the commands will be added to the bottom of the *File* menu.

986 #### Autostart Plugins
987 
988 This refers to a plugin which needs to be notified when ParaView starts up or
989 the plugin is loaded which ever happens later and then notified when ParaView
990 quits. Example is in `Examples/Plugins/Autostart` in the ParaView source. For
991 such a plugin, we need to provide a `QObject` subclass
992 (`pqMyApplicationStarter`) with methods that need to be called on startup and
993 shutdown.
994 
995 ```cpp
996 class pqMyApplicationStarter : public QObject
997 {
998 Q_OBJECT
999 public:
1000  // Callback for startup.
1001  // This cannot take any arguments
1002  void onStartup();
1003 
1004  // Callback for shutdown.
1005  // This cannot take any arguments
1006  void onShutdown();
1007 };
1008 ```
1009 
1010 The `CMakeLists.txt` looks as follows:
1011 
1012 ```cmake
1013 # Macro for auto-start plugins. We specify the class name and the methods to
1014 # call on startup and shutdown on an instance of that class. It returns the
1015 # interface and sources created in the variables passed to the `INTERFACES` and
1016 # `SOURCES` arguments, respectively.
1017 paraview_plugin_add_auto_start(
1018  CLASS_NAME pqMyApplicationStarter # the class name for our class
1019  STARTUP onStartup # specify the method to call on startup
1020  SHUTDOWN onShutdown # specify the method to call on shutdown
1021  INTERFACES interfaces
1022  SOURCES sources)
1023 
1024 # Create a plugin for this starter
1025 paraview_add_plugin(Autostart
1026  VERSION "1.0"
1027  UI_INTERFACES ${interfaces}
1028  SOURCES pqMyApplicationStarter.cxx ${interfaces})
1029 ```

1031 #### Adding new Representations for 3D View using Plugins
1032 
1033 ParaView's 3D view the most commonly used view for showing polygonal or
1034 volumetric data. By default, ParaView provides representation-types for showing
1035 the dataset as surface, wireframe, points etc. Its possible to add
1036 representations using plugins that extends this set of available
1037 representation types.
1038 
1039 Before we start looking at how to write such a plugin, we need to gain some
1040 understanding of the 3D view and its representations. The 3D view uses 3 basic
1041 representation proxies for rendering all types of data:
1042 
1043  * (representations, `UnstructuredGridRepresentation`) for
1044  `vtkUnstructuredGrid` or a composite dataset consisting of
1045  `vtkUnstructuredGrid`.
1046  * (representations, `UniformGridRepresentation`) for `vtkImageData` or a
1047  composite dataset consisting of `vtkImageData`
1048  * (representations, `GeometryRepresentation`) for all other data types.
1049 
1050 Each of these representation proxies are basically composite representation
1051 proxies that use other representation proxies to do the actual rendering, e.g.,
1052 `GeometryRepresentation` uses `SurfaceRepresentation` for rendering the data as
1053 wireframe, points, surface, and surface-with-edges and `OutlineRepresentation`
1054 for rendering an outline for the data. Subsequently, the 3 composite
1055 representation proxies provide a property named `Representation` which allows
1056 the user to pick the representation type he wants to see the data as. The
1057 composite representation proxy has logic to enable one of its internal
1058 representations based on the type chosen by the user.
1059 
1060 These 3 composite representation types are fixed and cannot be changed by
1061 plugins. What plugins can do is add more internal representations to any of
1062 these 3 composite representations to support new representations types that the
1063 user can choose using the representation type combo box on the display tab or
1064 in the toolbar.
1065 
1066 ![Representation type combo-box allowing user to choose the sub-representation to use](images/Representationplugin.png)
1067 
1068 ##### Using a New Mapper
1069 
1070 In this example, we see how to integrate a special polydata mapper written in
1071 VTK into ParaView. Lets say the mapper is called `vtkMySpecialPolyDataMapper`
1072 which is simply a subclass of `vtkPainterPolyDataMapper`. In practice,
1073 `vtkMySpecialPolyDataMapper` can internally use different painters to do
1074 perform special rendering tasks.
1075 
1076 To integrate this mapper into ParaView, first we need to create a
1077 `vtkSMRepresentationProxy` subclass for that uses this mapper. In this example,
1078 since the mapper is a simple replacement for the standard
1079 `vtkPainterPolyDataMapper`, we can define our representation proxy as a
1080 specialization of the `SurfaceRepresentation` as follows:
1081 
1082 ```xml
1083 <ServerManagerConfiguration>
1084  <ProxyGroup name="representations">
1085  <RepresentationProxy
1086  name="MySpecialRepresentation"
1087  class="vtkMySpecialRepresentation"
1088  processes="client|renderserver|dataserver"
1089  base_proxygroup="representations"
1090  base_proxyname="SurfaceRepresentation">
1091  <Documentation>
1092  This is the new representation type we are adding. This is identical to
1093  the SurfaceRepresentation except that we are overriding the mapper with
1094  our mapper.
1095  </Documentation>
1096  </RepresentationProxy>
1097  </ProxyGroup>
1098 </ServerManagerConfiguration>
1099 ```
1100 
1101 `vtkMySpecialRepresentation` is a subclass of
1102 `vtkGeometryRepresentationWithFaces` where in the constructor we simply
1103 override the mappers as follows:
1104 
1105 ```cpp
1106 vtkMySpecialRepresentation::vtkMySpecialRepresentation()
1107 {
1108  // Replace the mappers created by the superclass.
1109  this->Mapper->Delete();
1110  this->LODMapper->Delete();
1111 
1112  this->Mapper = vtkMySpecialPolyDataMapper::New();
1113  this->LODMapper = vtkMySpecialPolyDataMapper::New();
1114 
1115  // Since we replaced the mappers, we need to call SetupDefaults() to ensure
1116  // the pipelines are setup correctly.
1117  this->SetupDefaults();
1118 }
1119 ```
1120 
1121 Next we need to register this new type with the any (or all) of the 3 standard
1122 composite representations so that it will become available to the user to
1123 choose in the representation type combo box. To decide which of the 3 composite
1124 representations we want to add our representation to, think of the input data
1125 types our representation supports. If it can support any type of data set, then
1126 we can add our representation all the 3 representations (as is the case with
1127 this example). However if we are adding a representation for volume rendering
1128 of `vtkUnstructuredGrid` then we will add it only to the
1129 `UnstructuredGridRepresentation`. This is done by using the `Extension` XML
1130 tag. It simply means that we are extending the original XML for the proxy
1131 definition with the specified additions. Now to make this representation
1132 available as a type to the user, we use the `RepresentationType` element , with
1133 `text` used as the text shown for the type in the combo-box, `subproxy`
1134 specifies the name of representation subproxy to activate when the user chooses
1135 the specified type. Optionally one can also specify the `subtype` attribute,
1136 which if present is the value set on a property named `Representation` for the
1137 subproxy when the type is chosen. This allows for the subproxy to provide more
1138 than one representation type.
1139 
1140 ```xml
1141 <ServerManagerConfiguration>
1142  <ProxyGroup name="representations">
1143  <Extension name="GeometryRepresentation">
1144  <Documentation>
1145  Extends standard GeometryRepresentation by adding
1146  MySpecialRepresentation as a new type of representation.
1147  </Documentation>
1148 
1149  <!-- this adds to what is already defined in PVRepresentationBase -->
1150  <RepresentationType
1151  subproxy="MySpecialRepresentation"
1152  text="Special Mapper"
1153  subtype="1" />
1154 
1155  <SubProxy>
1156  <Proxy name="MySpecialRepresentation"
1157  proxygroup="representations"
1158  proxyname="MySpecialRepresentation">
1159  </Proxy>
1160  <ShareProperties subproxy="SurfaceRepresentation">
1161  <Exception name="Input" />
1162  <Exception name="Visibility" />
1163  <Exception name="Representation" />
1164  </ShareProperties>
1165  </SubProxy>
1166  </Extension>
1167  </ProxyGroup>
1168 </ServerManagerConfiguration>
1169 ```
1170 
1171 The `CMakeLists.txt` file is not much different from what it would be like for
1172 adding a simple filter or a reader where the representation class is placed
1173 into the contained module.
1174 
1175 Source code for this example is available under
1176 `Examples/Plugins/Representation` in the ParaView source directory.

1232 ## Plugins in Static Applications
1233 
1234 It is possible to import plugins into a ParaView-based application at compile
1235 time. When building ParaView-based applications statically, this is the only
1236 option to bring in components from plugins. When built statically (i.e., with
1237 `BUILD_SHARED_LIBS` set to false), ParaView will automatically link and load
1238 plugins that were enabled via CMake by inserting the necessary
1239 `PV_PLUGIN_IMPORT_INIT` and `PV_PLUGIN_IMPORT` macros.
1240 
1241 The code below shows how the `PV_PLUGIN` macros would be used to statically load
1242 plugins in custom applications:
1243 
1244 ```cpp
1245 #include "vtkPVPlugin.h"
1246 
1247 // Adds required forward declarations.
1248 PV_PLUGIN_IMPORT_INIT(MyFilterPlugin)
1249 PV_PLUGIN_IMPORT_INIT(MyReaderPlugin)
1250 
1251 class MyMainWindow : public QMainWindow
1252 {
1253  // ....
1254 };
1255 
1256 MyMainWindow::MyMainWindow(...)
1257 {
1258  // ... after initialization ...
1259 
1260  // Calls relevant callbacks to load the plugins and update the
1261  // GUI/Server-Manager
1262  PV_PLUGIN_IMPORT(MyFilterPlugin);
1263  PV_PLUGIN_IMPORT(MyReaderPlugin);
1264 }
1265 ```

1269 ### *Tools > Manage Plugins* is not visible!

1274 ### Compile error `invalid conversion from 'vtkYourFiltersSuperClass*' to 'vtkYourFilter*'`
1275 
1276 Any VTK object that needs to be treated as a filter or source has to be a
1277 `vtkAlgorithm` subclass. The particular superclass a filter is derived from has
1278 to be given not only in the standard C++ way:
1279 
1280 ```cpp
1281 class VTKMODULE_EXPORT vtkMyElevationFilter : public vtkElevationFilter
1282 ```
1283 
1284 but additionally declared with help of the `vtkTypeMacro`. For the example
1285 given above:
1286 
1287 ```cpp
1288 class VTKMODULE_EXPORT vtkMyElevationFilter : public vtkElevationFilter
1289 {
1290 public:
1291  vtkTypeMacro(vtkMyElevationFilter, vtkElevationFilter);
1292 }
1293 ```
1294 
1295 Otherwise, compiling the filter will fail with a variety of error messages
1296 (depending on superclass) like
1297 
1298 ```
1299 vtkMyElevationFilter.cxx:19: error: no 'void vtkMyElevationFilter::CollectRevisions(std::ostream&)'
1300  member function declared in class 'vtkMyElevationFilter'
1301 ```
1302 or
1303 
1304 ```
1305 vtkMyElevationFilterClientServer.cxx:97: error: invalid conversion from vtkPolyDataAlgorithm* to
1306  vtkICPFilter*
1307 ```

1309 ### Mysterious Segmentation Faults in Plugins that use Custom VTK Classes
1310 
1311 This primarily concerns plugins that make calls to your own custom `vtkMy` (or
1312 whatever you called it) library of VTK extensions.
1313 
1314 Symptoms:
1315 
1316  * The plugin will load, but causes a segfault when you try to use it.
1317  * If you use a debugger you may notice that in some cases when your code
1318  calls `vtkClassA.MethodB`, what actually gets called is
1319  `vtkClassC.MethodD`, where `MethodB` is a virtual member function. This is
1320  occurs because of different vtable entries in the Paraview-internal
1321  versions of the VTK libraries.
1322 
1323 The solution is to make sure that your `vtkMy` library is compiled against
1324 ParaView's internal VTK libraries. Even if you compiled VTK and ParaView using
1325 the same VTK sources, you *must not* link against the external VTK libraries.
1326 (The linker won't complain, because it will find all the symbols it needs, but
1327 this leads to unexpected behaviour.)
1328 
1329 To be explicit, when compiling your `vtkMy` library, you must set the CMake
1330 variable `VTK_DIR` to point to the `VTK` subdirectory in the directory in which
1331 you built ParaView. (On my system, CMake automatically finds VTK at
1332 `/usr/lib/vtk-5.2`, and I must change `VTK_DIR` to
1333 `~/source/ParaView3/build/VTK`.)

1336

1338 in doubt, try placing your plugin and all its dependent DLLs in the `bin`
1339 directory of your build and load it from there.
