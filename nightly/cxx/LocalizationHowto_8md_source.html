2 
3 ## Introduction
4 
5 ParaView comes with a localization system that handles the complete language introspection workflow, from translation searching in the source code to language selection in the settings.
6 
7 The theoretical workflow for a single language runs as follows:
8 
9 - The ParaView build system generates translation source files (.ts files) in the chosen destination repository.
10 - The translation from English to the desired language is written into these source files
11 - These translations files are compiled into a binary file (.qm file) and installed in the `translations` shared folder of ParaView.
12 - The language can then be selected in the settings `Interface Language` section.
13 
14 In practice, the process is slightly more complex since we use an official translation source repository that is updated by a translations CMS, for simplifying contributions.
15 
16 ### Technologies used
17 
18 Most of the steps are done using two of the [Qt5's linguist utilities](https://doc.qt.io/qt-5/linguist-manager.html):
19 - `lupdate`, a source file parser that can parse C++ and Qt's UI files for translatable strings, and then generates a .ts file from it
20 - `lrelease`, a .ts file compiler, that generates the .qm files.
21 
22 These utilities are invoked in the ParaView CMake build system and in the official ts repository respectively.
23 
24 ### What is translated
25 
26 - Interface (Menus, buttons...)
27 - Proxy and property names/labels and docs (see below for the exhaustive list).
28 
29 ## Glossary
30 
31 - locale: In this context, a locale is a string in the format `<language>_<COUNTRY>`. It is a language's national variant.
32 - lrelease: A Qt utility that transforms ts files into a qm.
33 - lupdate: A Qt utility that parses .ui and C++ files into ts files.
34 - .qm: A Qt resource file containing translations.
35 - Qt linguist: A Qt tool to fill ts files with translations.
36 - translation context: Qt enforces that all translations are within a context, possibly user-specified. In ParaView, the standard is to use the class name as context. For a translation to be used, the context must be the same at parse-time and at runtime.
37 - .ts: A Qt's xml containing string to translate and its translation if filled.
38 - .ui: A Qt resource file containing user interface layouts.

40 ## Writing translatable source code
41 
42 All the source code in ParaView should ideally be translatable. A string is _translatable_ if it can be found by lupdate.
43 
44 Strings displayed in the ParaView client can come from three main sources:
45 - Qt's UI files
46 - C++ source code that is Qt-related
47 - XML proxy files
48 
49 Qt's UI files are automatically found by lupdate, so if a string shall **not** be translated (ie ParaView's name !) it has to have its `translatable` property disabled in Qt Designer.
50 
51 ### In the ParaView Qt classes
52 
53 An exhaustive guide to translatable source code using Qt5 can be found [here](https://doc.qt.io/qt-5/i18n-source-translation.html).
54 
55 For setting the translatable property of a string in the most common cases:
56 - If an explicit string is in a class inheriting QObject that has the `Q_OBJECT` macro, you can use the `tr(<explicitStringToTranslate>)` method of `QObject`, without specifying `this->` before ([example](https://gitlab.kitware.com/paraview/paraview/-/blob/c055918678939470ba25e2334e131870da8f2fde/Qt/ApplicationComponents/pqSaveDataReaction.cxx#L159)).
57 - If a class cannot be given the `Q_OBJECT` macro, or does not inherit from `QObject`, you can either:
58  - add the `Q_DECLARE_TR_FUNCTIONS(<className>)` to the class and then call `tr(<explicitStringToTranslate>)` inside ([example](https://gitlab.kitware.com/paraview/paraview/-/blob/c055918678939470ba25e2334e131870da8f2fde/Qt/Components/pqArrayListWidget.cxx#L46)) or
59  - write `QCoreApplication::translate(<className>, <explicitStringToTranslate>)` for every string to translate in the class ([example](https://gitlab.kitware.com/paraview/paraview/-/blob/c055918678939470ba25e2334e131870da8f2fde/Qt/ApplicationComponents/pqParaViewMenuBuilders.cxx#L329)).
60 - If the string is located outside of a class, method or even function, you can use `QT_TRANSLATE_NOOP(<contextName>, <explicitStringToTranslate>)`.
61 
62 All these methods requires the include of `QCoreApplication`.
63 
64 Some examples:
65 ```cpp
66 // When inside a class method with Q_OBJECT macro
67 class someClass : QObject
68 {
69  Q_OBJECT
70  QString someMethod();
71 };
72 
73 QString someClass::someMethod()
74 {
75  return tr("Translated string");
76 }
77 
78 // When inside a class without Q_OBJECT macro
79 class someNonQtClass
80 {
81  Q_DECLARE_TR_FUNCTIONS(someNonQtClass)
82  QString someMethod();
83 };
84 
85 QString someNonQtClass::someMethod()
86 {
87  return tr("Translated string");
88 }
89 
90 class someNonQtClass2
91 {
92  QString someMethod();
93 };
94 
95 QString someNonQtClass2::someMethod()
96 {
97  return QCoreApplication::translate("someNonQtClass2", "Translated string");
98 }
99 
100 // When outside of any method
101 QString someFunction()
102 {
103  return QT_TRANSLATE_NOOP("translationContext", "Translated string");
104 }
105 ```

107 ### Making Proxy XML elements translatable
108 
109 The XML format is not natively supported by lupdate. Therefore, `CMake/XML_translations_header_generator.py` was developed. It transforms proxy XMLs into C++ headers in order to be found by lupdate.
110 
111 In a C++ code that displays text from the XML, it is now mandatory to use the context "ServerManagerXML" so that the translation can be found at runtime. E.g.: `QCoreApplication::translate("ServerManagerXML", <translated XML element>));`.
112 
113 Currently translatable XML fields are:
114 
115 - `label`, which can be an attribute of any element
116 - `menu_label`, which can be an attribute of any element
117 - `Documentation` element's content
118  - `long_help` attribute
119  - `short_help` attribute
120 - All `*Property`'s `name`s
121 
122 This behavior can be extended by adding new elements or attributes to parse in the python utility.

125 ## Testing new translations
126 
127 The plugin `TranslationsTesting`, bundled with ParaView, is built by default when enabling the `PARAVIEW_BUILD_TRANSLATIONS` option. It will search though the main window for untranslated strings and print a warning if found.
128 
129 This plugin is meant to be run through ctest, with the `TranslationsTesting` test. It might be slow since it uses Regexes for its widget-ignore list (hopefully this list will be empty, otherwise, you will have the time to admire the improved version of ParaView).

132 ### Ignoring untranslated strings
133 
134 Some source strings could not, or should not be translatable (ie ParaView's name !).
135 ParaView translatability is tested on the ParaView's Gitlab CI, so if a string cannot be translated, the widget/action containing it **has** to be be added on the ignore list of the plugin.
136 
137 The ignore list is located at `Plugins/TranslationsTesting/pqTranslationsTesting.h`. You can use either
138 - `TRANSLATION_IGNORE_STRINGS` to ignore a precise widget/action, or
139 - `TRANSLATION_IGNORE_REGEXES` to ignore a range of widgets.
140 
141 Both lists take pairs of strings: the first element is the name of the widget and the second is the attribute to ignore or an empty string to ignore all attributes.
142 
143 The tested Qt attributes are text, `toolTip`, `windowTitle`, `placeholderText`, `text` and `label`.

145 ## Translation workflow
146 
147 ### Source files generation
148 
149 When `PARAVIEW_BUILD_TRANSLATIONS` is enabled, translations source files are built in the given `PARAVIEW_TRANSLATIONS_DIRECTORY`. A specific build target `localization` can be used separatly from the ALL target to build them.
150 
151 ### Source files filling
152 There is an official [paraview-translations](https://gitlab.kitware.com/paraview/paraview-translations) repository containing up-to-date source files.
153 
154 The target `files_update` will complete all ts files of all languages with the source strings. A single language can also be updated using the special target `<locale>_update`.
155 
156 After this step, translators can fill ts files with translations using [Qt Linguist](https://doc.qt.io/qt-5/linguist-translators.html) or any other tool supporting ts files.
157 
158 ### Source file compilation
159 
160 Source files can be compiled into qm files using the CMake process of the translations repository. Each locale with a folder in the translations repository is a valid target that will create a qm.
161 
162 It is also possible to compile ts files manually using lrelease. The command is `lrelease <input files> -qm <outputQm>`.
163 
164 ### Binary files loading
165 
166 Qm files can be loaded from the general settings. If a qm is found by ParaView, the language it provides will be available in the `Interface language` Combobox. The selected language is saved in the settings, and a restart is required.
167 
168 By default, ParaView searches for qms in the `translations` shared folder. The environment variable `PV_TRANSLATIONS_DIR` can provide other searching paths. When multiple qm files have the same language, the first qm found will be used, using the `PV_TRANSLATIONS_DIR` paths first, and the default path after.
169 
170 The locale can also be forced using the `PV_TRANSLATIONS_LOCALE` environment variable.
