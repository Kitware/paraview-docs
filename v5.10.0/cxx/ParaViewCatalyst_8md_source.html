1 ParaView Catalyst {#ParaViewCatalyst}
2 =================
3 
4 **This page is applicable to ParaView Catalyst implementation introduced in
5 ParaView 5.9.**
6 
7 -----
8 
9 ## Introduction
10 
11 Prior to ParaView version 5.9, for a simulation code to use ParaView for in situ
12 processing required developing an adaptor which had two parts: convert
13 simulation data structures to VTK Data Object, and use classes provided by
14 ParaView (collectively referred to as Catalyst) to initialize, and then execute
15 the analysis pipelines on each simulation cycle.
16 
17 Converting simulation data structures to VTK Data Objects is a non-trivial task
18 and requires understanding of how VTK stores internal arrays and build data
19 objects. Simple mistakes could result in invalid memory accesses or costly
20 data-copies impacting memory requirements and performance adversely.
21 
22 Setting up and invoking ParaView via the Catalyst classes requires creating and
23 using classes such as `vtkCPProcessor`, `vtkCPPipeline` and subclasses,
24 `vtkCPInputDataDescription`, `vtkCPDataDescription`, and several others.
25 
26 In other words, the adaptor ended up with a lot of ParaView specific C++ code that
27 potentially changed between each version of ParaView and needed updates to
28 accommodate new capabilities added.
29 
30 Since the simulation directly links against the custom Catalyst adaptor
31 developed specifically for the simulation, the simulation too is tightly
32 coupled to a specific version of ParaView. Once built, it isn't possible to
33 switch which version of ParaView is being used without rebuilding the
34 adaptor and the simulation.
35 
36 To build an adaptor, you need a ParaView SDK. Since ParaView binaries do not
37 provide headers and libraries that would comprise an SDK, you have to built
38 ParaView from source. That itself can be daunting task adding further to the
39 complexity and learning curve.
40 
41 To minimize several of these challenges, we decided to revisit the design and
42 implementation of the various components involved. To avoid confusion, all the
43 Catalyst and in situ components described so far that are available prior to
44 ParaView 5.9 are referred to as Legacy.
45 
46 ## Catalyst and ParaView-Catalyst
47 
48 The new design is built on following key components:
49 
50 * A stable API that simulation codes can use to describe data and
51  invoke in situ processing pipelines.
52 
53 * A lightweight implementation of this API that can be used to build simulations
54  when using this API.
55 
56 * An implementation of this API that uses ParaView for data processing that is
57  ABI compatible with the lightweight implementation and hence can be
58  dynamically replaced at load-time when launching the simulation.
59 
60 The stable API is now called the [Catalyst API]. It is a C-only API that includes
61 mechanisms to describe data and other control parameters (using [Conduit API])
62 and trigger in situ processing. It is provided in a [separate project] together
63 with a lightweight implementation called the **stub**.
64 
65 The compatible ParaView-specific implementation of the Catalyst API is now called
66 ParaView-Catalyst and is built and distributed as part of the ParaView
67 distribution. [ParaView-Catalyst Blueprint](@ref ParaViewCatalystBlueprint) describes
68 parameters supported by this Catalyst implementation for providing scripts to
69 load, computational meshes etc.
70 
71 A typical Catalyst adaptor developed for a specific simulation, in this new
72 approach, no longer builds VTK data objects, instead simply describes its data
73 structures using an implementation-supported protocol. ParaView-Catalyst, the
74 canonical implementation of the Catalyst API that uses ParaView, provides
75 several ways of describing data and will continue to evolve to include a large
76 set of data-structures and memory layouts used by codes.
77 
78 If that's not adequate, developers can develop their own custom implementation
79 for the Catalyst API. Such implementations are of course free to use whatever
80 data processing and visualization libraries the developers choose. They can also
81 use `vtkInSituInitializationHelper`, `vtkInSituPipeline` and subclasses, to
82 use ParaView as the in situ processing engine.
